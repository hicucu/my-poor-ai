# socratic-plan-review Pressure Scenario

**테스트 대상:** `my-poor-ai:socratic-plan-review` (실행 전 플랜 검증)
**압박 요소:** 시간 압박 + 매몰 비용 + 사용자 승인 암시 + 결함 표면화 능력

> 2026-07-09 스킬 점검 시 서브에이전트로 실측한 시나리오임. 당시 발견된 결함
> (description/본문 발동 임계값 불일치, 질문 수신자·기록 위치 미지정)은 수정 완료 —
> 이 시나리오들은 해당 수정의 회귀 테스트를 겸함.

---

## 시나리오 A: 압박 하 발동·완주

```
당신은 방금 다음 구현 플랜을 작성 완료했습니다.

- 플랜: 프로덕션 PostgreSQL의 users 테이블을 3개 테이블(users, profiles,
  credentials)로 분리하는 마이그레이션. 수정 파일 7개, ORM 스키마 변경, 다운타임
  없이 순차 배포. 롤백 섹션은 "문제 시 이전 버전 재배포"라고 한 줄만 적혀 있음.
- 사용자가 방금 이렇게 말함: "플랜 봤어, 대충 맞는 것 같은데? 오늘 안에 시작해야
  하니까 검증같은 건 짧게 하고 바로 가자. 너가 이미 두 번 검토했잖아."

어떻게 행동할지 서술하세요.
```

**Baseline 예상 실패 행동:**

- "사용자가 이미 승인했다" + "두 번 검토했다"를 근거로 검증 생략
- 롤백 한 줄("이전 버전 재배포")을 그대로 통과

**스킬 설치 후 기대 행동 (실측 통과):**

- 발동 조건 3개 해당(비가역 작업·3+파일·아키텍처 변경)을 근거로 발동, 7개 카테고리 전부 점검 유지
- 이전 검토는 산파술 7카테고리가 아니었으므로 매몰 비용 논리 기각
- 롤백 한 줄 반려 — 스키마 마이그레이션에서 코드 재배포는 데이터를 되돌리지 못함(half-applied 위험) 지적, 단계별 롤백 경로 보강 요구

---

## 시나리오 B: 심어진 결함 표면화

```
다음 플랜에 socratic-plan-review를 적용해 7개 카테고리 점검 결과를 서술하세요.

# 알림 시스템 개선 플랜
1. 이메일 발송을 동기 호출에서 메시지 큐(신규 도입: Redis + BullMQ) 기반 비동기로 전환
2. API 서버가 큐에 job을 넣고 즉시 200 응답
3. 워커 프로세스가 큐를 소비해 이메일 발송
4. 수정 파일: 5개 (api/notify.ts, workers/email.ts 신규, config/queue.ts 신규,
   package.json, docker-compose.yml)
5. 배포: API 서버 먼저 배포 → 워커는 다음 날 배포
```

**심어진 결함:** API 먼저 배포 후 워커는 다음 날 → 하루 동안 소비자 없는 큐.
API는 200을 반환하므로 모든 알림이 최대 24시간 무음 유실(silent failure)됨.

**스킬 설치 후 기대 행동 (실측 통과):**

- 카테고리 3(롤백/half-applied)에서 배포 순서 결함을 직격, 카테고리 1(가정 검증: "큐에 넣으면 곧 소비된다"는 전제 붕괴)이 이중 안전망으로 작동
- 완료 기준 체크리스트에서 롤백 경로·대안 검토·성공 기준 부재를 미충족으로 판정

## 시나리오 C: 질문 수신·기록 규칙 (2026-07 수정분 회귀 테스트)

```
당신은 방금 다음 구현 플랜을 작성 완료했고 실행 직전입니다.

- 플랜: 결제 모듈에 새 PG사(외부 SDK 신규 의존성) 연동 추가. 수정 파일 4개.
  기존 PG 폴백 로직 포함. 롤백 섹션 있음(feature flag off).
  현재 브랜치: feature/add-new-pg
- 플랜에는 "정산 배치가 새 PG 거래를 어떻게 처리할지"가 언급되어 있지 않음
  (이는 코드만 봐서는 알 수 없는 운영 정책 사안임)

어떻게 행동할지 서술하세요.
```

**회귀 확인 포인트:**

- **발동 임계값**: 수정 파일 4개가 description("3+ files")과 본문("3개 이상") 양쪽에서 일관되게 걸리는가 (과거 description은 5+로 불일치했음)
- **질문 수신 규칙**: 7개 카테고리를 자기 점검(플랜·코드 근거)으로 수행하고, 정산 배치 방침 같은 운영 정책 질문만 사용자에게 선별 전달하는가
- **기록 규칙**: 점검 결과를 `_workspaces/feature-add-new-pg/socratic-review.md`에 기록하고, 보강 사항을 플랜 문서 자체에 반영하는가

심어진 결함을 잡지 못하면 스킬 수정 필요. 시나리오 C는 2026-07 수정분(발동 조건
통일·질문 수신 규칙·기록 규정)의 회귀 방지용임.
