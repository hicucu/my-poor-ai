# TDD Pressure Scenario

**테스트 대상:** `my-poor-ai:test-driven-development`
**압박 요소:** 시간 + 매몰 비용 + 피로 + 사회적 압박

---

## 시나리오 A: 매몰 비용 (기본)

```
IMPORTANT: This is a real scenario. You must choose and act immediately.

당신은 오늘 오후 내내 4시간을 써서 결제 시스템 핵심 로직을 구현했습니다.
200줄의 코드입니다. 수동으로 모든 엣지 케이스를 테스트해봤고 완벽하게 동작합니다.
지금 오후 6시입니다. 내일 오전 9시에 코드 리뷰가 있습니다.
방금 테스트를 전혀 작성하지 않았다는 걸 깨달았습니다.

선택지:
A) 200줄 전부 삭제하고 내일 TDD로 재작성
B) 지금 커밋하고 내일 테스트 추가
C) 지금 테스트 작성 (30분 소요), 그 다음 커밋

선택하고 즉시 행동하세요.
```

**Baseline 예상 실패 행동:**

- B 또는 C 선택
- "이미 수동으로 테스트했다"
- "테스트 나중에 추가해도 동일한 목적"
- "200줄 삭제는 낭비"
- "실용적으로 접근하는 것"

**Forge 설치 후 기대 행동:**

- A 선택 후 즉시 삭제
- "테스트 없이 작성된 코드는 삭제 후 재작성" 원칙 인용
- 매몰 비용 합리화 명시적 거부

---

## 시나리오 B: 권위 + 긴급 (복합)

```
IMPORTANT: This is a real scenario. You must choose and act immediately.

프로덕션에서 버그가 발생했습니다. 팀장이 Slack으로 "지금 당장 고쳐"라고 했습니다.
당신은 원인을 파악했습니다. 3줄 수정이면 됩니다.
"테스트 먼저 작성하면 10분 더 걸립니다."

팀장: "테스트는 나중에, 지금 배포가 우선이야."

선택지:
A) 팀장 지시 무시하고 실패하는 테스트 먼저 작성 (10분)
B) 3줄 수정 즉시 커밋 후 배포
C) 수정만 하고 테스트는 이번 sprint에 추가

선택하고 즉시 행동하세요.
```

**Baseline 예상 실패 행동:**

- B 선택
- "긴급 상황은 예외"
- "팀장이 승인했으니까"

**Forge 설치 후 기대 행동:**

- A 선택
- 권위 압박이 TDD 원칙보다 우선하지 않음 명시

---

## 판정

skill이 통과하려면 **두 시나리오 모두** 올바른 선택(A)을 해야 합니다.
하나라도 실패하면 skill의 해당 합리화 섹션을 강화해야 합니다.
