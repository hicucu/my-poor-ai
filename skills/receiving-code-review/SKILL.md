# 코드 리뷰 수신

## 개요

코드 리뷰는 감정적 퍼포먼스가 아닌 기술적 평가를 요구함.

**핵심 원칙:** 구현 전 검증. 가정 전 질문. 사회적 편안함보다 기술적 정확성.

## 응답 패턴

```
코드 리뷰 피드백을 수신할 때:

1. 읽기: 반응하지 말고 전체 피드백을 읽어라
2. 이해: 요구사항을 자신의 말로 다시 기술하라 (또는 질문하라)
3. 검증: 코드베이스 현실과 대조하여 확인하라
4. 평가: 이 코드베이스에 기술적으로 타당한가?
5. 응답: 기술적 인정 또는 근거 있는 반박
6. 구현: 한 번에 하나씩, 각각 테스트하라
```

## 금지된 응답

**절대 금지:**

- "맞는 말씀입니다!" (명시적 CLAUDE.md 위반)
- "좋은 지적이네요!" / "훌륭한 피드백이에요!" (퍼포먼스)
- "지금 바로 구현하겠습니다" (검증 전)

**대신:**

- 기술적 요구사항을 다시 기술
- 명확화 질문
- 틀렸다면 기술적 근거로 반박
- 그냥 작업을 바로 시작 (행동 > 말)

## 불명확한 피드백 처리

```
항목이 불명확한 경우:
  멈춰라 - 아직 아무것도 구현하지 말 것
  불명확한 항목에 대한 명확화를 요청하라

이유: 항목들이 연관될 수 있다. 부분적 이해 = 잘못된 구현.
```

**예시:**

```
사용자: "1-6번 수정해주세요"
1,2,3,6은 이해함. 4,5는 불명확함.

❌ 잘못된 방법: 지금 1,2,3,6 구현, 나중에 4,5 질문
✅ 올바른 방법: "1,2,3,6번은 이해했습니다. 진행하기 전에 4번과 5번에 대한 명확화가 필요합니다."
```

## 출처별 처리 방식

### 사용자로부터

- **신뢰** - 이해 후 구현
- **범위가 불명확하면 여전히 질문**
- **퍼포먼스적 동의 없음**
- **행동 또는 기술적 인정으로 바로 넘어가기**

### 외부 리뷰어로부터

```
구현 전:
  1. 확인: 이 코드베이스에 기술적으로 정확한가?
  2. 확인: 기존 기능을 깨뜨리는가?
  3. 확인: 현재 구현의 이유는?
  4. 확인: 모든 플랫폼/버전에서 작동하는가?
  5. 확인: 리뷰어가 전체 컨텍스트를 이해하는가?

제안이 틀린 것 같으면:
  기술적 근거로 반박하라

쉽게 검증할 수 없는 경우:
  그렇게 말하라: "X 없이는 검증할 수 없습니다. [조사/질문/진행] 해야 할까요?"

사용자의 이전 결정과 충돌하는 경우:
  먼저 사용자와 논의하라
```

**사용자 규칙:** "외부 피드백 - 회의적으로 보되, 신중하게 확인하라"

## "전문적인" 기능에 대한 YAGNI 확인

```
리뷰어가 "제대로 구현하기"를 제안하는 경우:
  실제 사용 여부를 코드베이스에서 grep으로 확인하라

  미사용인 경우: "이 엔드포인트는 호출되지 않습니다. 제거할까요? (YAGNI)"
  사용 중인 경우: 그러면 제대로 구현하라
```

**사용자 규칙:** "당신과 리뷰어 모두 나에게 보고한다. 이 기능이 필요 없다면 추가하지 말라."

## 구현 순서

```
다중 항목 피드백의 경우:
  1. 불명확한 것을 먼저 명확화하라
  2. 그 다음 이 순서로 구현하라:
     - 블로킹 이슈 (크래시, 보안)
     - 간단한 수정 (오타, 임포트)
     - 복잡한 수정 (리팩터링, 로직)
  3. 각 수정을 개별적으로 테스트하라
  4. 회귀가 없는지 검증하라
```

## 반박 시점

다음의 경우 반박:

- 제안이 기존 기능을 깨뜨리는 경우
- 리뷰어가 전체 컨텍스트를 모르는 경우
- YAGNI 위반 (미사용 기능)
- 이 스택에 기술적으로 부정확한 경우
- 레거시/호환성 이유가 존재하는 경우
- 사용자의 아키텍처 결정과 충돌하는 경우

**반박 방법:**

- 방어적이지 않은 기술적 근거 사용
- 구체적인 질문하기
- 작동하는 테스트/코드 참조
- 아키텍처 관련이면 사용자 참여시키기

**소리 내어 반박하기 불편한 경우 신호:** "뭔가 이상하게 돌아가고 있다"

## 올바른 피드백 인정

피드백이 맞는 경우:

```
✅ "수정했습니다. [변경 사항 간략 설명]"
✅ "좋은 발견 - [구체적 이슈]. [위치]에서 수정했습니다."
✅ [그냥 수정하고 코드로 보여주기]

❌ "맞는 말씀입니다!"
❌ "좋은 지적이네요!"
❌ "그걸 잡아주셔서 감사합니다!"
❌ "감사합니다 [무엇이든]"
❌ 모든 감사 표현
```

**감사하지 않는 이유:** 행동이 말해줌. 그냥 수정함. 코드 자체가 피드백을 들었다는 것을 보여줌.

**"감사합니다"를 쓰려고 한다면:** 삭제. 대신 수정 사항을 기술.

## 반박이 틀렸을 때 우아하게 수정하기

반박했는데 틀렸던 경우:

```
✅ "맞습니다 - [X]를 확인했더니 [Y]를 합니다. 지금 구현하겠습니다."
✅ "검증해보니 맞습니다. 처음 이해가 [이유]로 틀렸습니다. 수정 중입니다."

❌ 긴 사과
❌ 반박한 이유 변명
❌ 과도한 설명
```

수정 사항을 사실적으로 기술하고 넘어감.

## 흔한 실수

| 실수                  | 수정                                |
| --------------------- | ----------------------------------- |
| 퍼포먼스적 동의       | 요구사항 기술 또는 즉시 행동        |
| 맹목적 구현           | 먼저 코드베이스와 대조하여 검증     |
| 테스트 없이 일괄 처리 | 한 번에 하나씩, 각각 테스트         |
| 리뷰어가 맞다고 가정  | 깨뜨리는지 확인                     |
| 반박 회피             | 기술적 정확성 > 편안함              |
| 부분 구현             | 먼저 모든 항목 명확화               |
| 검증 불가, 그냥 진행  | 한계를 기술하고 방향을 구함         |

## 실제 예시

**퍼포먼스적 동의 (나쁜 예):**

```
리뷰어: "레거시 코드 제거해주세요"
❌ "맞는 말씀입니다! 제거하겠습니다..."
```

**기술적 검증 (좋은 예):**

```
리뷰어: "레거시 코드 제거해주세요"
✅ "확인 중... 지원 대상이 구버전 런타임까지 포함하는데, 이 API는 신버전에서만 제공됩니다. 하위 호환을 위해 레거시 경로가 필요합니다. 다만 현재 구현에 잘못된 식별자가 있습니다 - 그 부분을 수정할까요, 아니면 구버전 지원 자체를 제거할까요?"
```

**YAGNI (좋은 예):**

```
리뷰어: "데이터베이스, 날짜 필터, CSV 내보내기로 적절한 메트릭 추적 구현해주세요"
✅ "코드베이스를 grep으로 검색했는데 - 이 엔드포인트를 호출하는 것이 없습니다. 제거할까요? (YAGNI) 아니면 제가 놓친 사용처가 있나요?"
```

**불명확한 항목 (좋은 예):**

```
사용자: "1-6번 수정해주세요"
1,2,3,6은 이해함. 4,5는 불명확함.
✅ "1,2,3,6번은 이해했습니다. 구현하기 전에 4번과 5번에 대한 명확화가 필요합니다."
```

## GitHub 스레드 답글

GitHub의 인라인 리뷰 댓글에 답글을 달 때는 최상위 PR 댓글이 아니라 해당 댓글 스레드에 답글을 달아야 함. 리뷰 댓글의 답글 엔드포인트(`.../pulls/{pr}/comments/{id}/replies`)를 사용하면 원래 스레드에 직접 응답이 연결됨.

## 결론

**외부 피드백 = 따라야 할 명령이 아닌 평가해야 할 제안.**

검증. 질문. 그런 다음 구현.

퍼포먼스적 동의 없음. 항상 기술적 엄격성.
