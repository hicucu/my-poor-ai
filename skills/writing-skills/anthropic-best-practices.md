# 스킬 작성 모범 사례

> Claude가 발견하고 성공적으로 사용할 수 있는 효과적인 스킬을 작성하는 법.

> 출처: 이 문서는 Anthropic의 공식 Agent Skills 작성 가이드를 한국어로 옮기고, 특정 언어/프레임워크에 묶인 예시를 스택 중립적으로 일반화한 것임.

좋은 스킬은 간결하고, 잘 구조화돼 있으며, 실제 사용으로 테스트됨. 이 가이드는 Claude가 발견하고 효과적으로 사용할 수 있는 스킬을 작성하기 위한 실질적인 결정들을 제공함.

스킬의 작동 원리에 대한 개념적 배경은 Agent Skills 개요 문서를 참조.

## 핵심 원칙

### 간결함이 핵심

[컨텍스트 윈도우](https://platform.claude.com/docs/en/build-with-claude/context-windows)는 공공재임. 당신의 스킬은 Claude가 알아야 할 다른 모든 것과 컨텍스트 윈도우를 공유함:

- 시스템 프롬프트
- 대화 기록
- 다른 스킬의 메타데이터
- 실제 요청

스킬의 모든 토큰에 즉각적인 비용이 드는 것은 아님. 시작 시점에는 모든 스킬의 메타데이터(name과 description)만 미리 로드됨. Claude는 스킬이 관련될 때만 SKILL.md를 읽고, 필요할 때만 추가 파일을 읽음. 그래도 SKILL.md를 간결하게 유지하는 것은 여전히 중요함: 일단 Claude가 로드하면, 모든 토큰이 대화 기록 및 다른 컨텍스트와 경쟁함.

**기본 가정:** Claude는 이미 매우 똑똑함.

Claude가 아직 갖고 있지 않은 컨텍스트만 추가할 것. 모든 정보 조각에 의문을 제기할 것:

- "Claude가 정말 이 설명을 필요로 하는가?"
- "Claude가 이것을 안다고 가정할 수 있는가?"
- "이 문단이 토큰 비용을 정당화하는가?"

**좋은 예시: 간결함** (약 50 토큰):

````markdown
## 문서에서 텍스트 추출

표준 추출 라이브러리로 텍스트를 추출한다:

```
문서 열기 → 페이지 선택 → 텍스트 추출
```
````

**나쁜 예시: 너무 장황함** (약 150 토큰):

```markdown
## 문서에서 텍스트 추출

문서 파일은 텍스트, 이미지, 기타 콘텐츠를 담는 흔한 파일 형식입니다.
문서에서 텍스트를 추출하려면 라이브러리가 필요합니다. 문서 처리를 위한
많은 라이브러리가 있지만, 사용하기 쉽고 대부분의 경우를 잘 처리하기 때문에
특정 라이브러리를 권장합니다. 먼저 패키지 매니저로 설치해야 합니다.
그다음 아래 코드를 사용할 수 있습니다...
```

간결한 버전은 Claude가 문서가 무엇이고 라이브러리가 어떻게 작동하는지 안다고 가정함.

### 적절한 자유도 설정

작업의 취약성과 가변성에 맞춰 구체성의 수준을 조정할 것.

**높은 자유도** (텍스트 기반 지시):

사용 시점:

- 여러 접근법이 유효함
- 결정이 맥락에 의존함
- 휴리스틱이 접근을 안내함

예시:

```markdown
## 코드 리뷰 프로세스

1. 코드 구조와 구성을 분석
2. 잠재적 버그나 엣지 케이스 확인
3. 가독성과 유지보수성 개선 제안
4. 프로젝트 관례 준수 여부 검증
```

**중간 자유도** (매개변수가 있는 의사코드나 스크립트):

사용 시점:

- 선호하는 패턴이 존재함
- 일부 변형이 허용됨
- 설정이 동작에 영향을 줌

예시:

````markdown
## 보고서 생성

이 템플릿을 사용하고 필요에 따라 조정한다:

```
보고서_생성(데이터, 형식="markdown", 차트_포함=true):
    # 데이터 처리
    # 지정된 형식으로 출력 생성
    # 선택적으로 시각화 포함
```
````

**낮은 자유도** (구체적인 스크립트, 매개변수가 거의 없거나 없음):

사용 시점:

- 작업이 취약하고 오류가 발생하기 쉬움
- 일관성이 중요함
- 특정 순서를 반드시 따라야 함

예시:

````markdown
## 데이터베이스 마이그레이션

정확히 이 스크립트를 실행한다:

```
마이그레이션 스크립트를 --verify --backup 옵션과 함께 실행
```

명령을 수정하거나 추가 플래그를 붙이지 말 것.
````

**비유:** Claude를 경로를 탐험하는 로봇이라고 생각할 것:

- **양쪽이 절벽인 좁은 다리:** 앞으로 나아갈 안전한 길은 단 하나뿐임. 구체적인 가드레일과 정확한 지시를 제공하라(낮은 자유도). 예: 정확한 순서로 실행해야 하는 데이터베이스 마이그레이션.
- **위험이 없는 열린 들판:** 많은 길이 성공으로 이어짐. 큰 방향만 제시하고 Claude가 최선의 경로를 찾도록 신뢰하라(높은 자유도). 예: 맥락이 최선의 접근을 결정하는 코드 리뷰.

### 사용할 모든 모델로 테스트

스킬은 모델에 대한 추가물로 작동하므로, 효과는 기반 모델에 달려 있음. 사용하려는 모든 모델로 스킬을 테스트할 것.

**모델별 테스트 고려사항:**

- **Claude Haiku** (빠르고 경제적): 스킬이 충분한 안내를 제공하는가?
- **Claude Sonnet** (균형): 스킬이 명확하고 효율적인가?
- **Claude Opus** (강력한 추론): 스킬이 과도한 설명을 피하는가?

Opus에 완벽한 것이 Haiku에는 더 많은 세부사항이 필요할 수 있음. 여러 모델에서 스킬을 사용할 계획이라면, 그 모두에서 잘 작동하는 지시를 목표로 할 것.

## 스킬 구조

> **YAML Frontmatter:** SKILL.md frontmatter는 두 필드를 필요로 함:
>
> - `name` — 스킬의 사람이 읽을 수 있는 이름 (최대 64자)
> - `description` — 스킬이 무엇을 하고 언제 사용하는지에 대한 한 줄 설명 (최대 1024자)
>
> 완전한 스킬 구조 세부사항은 Agent Skills 개요 문서를 참조.

### 이름 짓기 관례

스킬을 참조하고 논의하기 쉽도록 일관된 이름 패턴을 사용할 것. 스킬 이름에는 **동명사 형태**(동사 + -ing)를 권장하는데, 스킬이 제공하는 활동이나 능력을 명확히 설명하기 때문임.

**좋은 이름 예시 (동명사 형태):**

- "Processing PDFs"
- "Analyzing spreadsheets"
- "Managing databases"
- "Testing code"
- "Writing documentation"

**허용되는 대안:**

- 명사구: "PDF Processing", "Spreadsheet Analysis"
- 행동 지향: "Process PDFs", "Analyze Spreadsheets"

**피할 것:**

- 모호한 이름: "Helper", "Utils", "Tools"
- 지나치게 일반적: "Documents", "Data", "Files"
- 스킬 모음 내에서 일관되지 않은 패턴

일관된 이름 짓기는 다음을 쉽게 만듦:

- 문서와 대화에서 스킬 참조
- 한눈에 스킬이 하는 일 이해
- 여러 스킬을 정리하고 검색
- 전문적이고 응집력 있는 스킬 라이브러리 유지

### 효과적인 description 작성

`description` 필드는 스킬 발견을 가능하게 하며, 스킬이 *무엇을 하는지*와 _언제 사용하는지_ 둘 다 포함해야 함.

> **반드시 3인칭으로 작성할 것.** description은 시스템 프롬프트에 주입되며, 시점이 일관되지 않으면 발견 문제를 일으킬 수 있음.
>
> - **좋음:** "Processes Excel files and generates reports"
> - **피할 것:** "I can help you process Excel files"
> - **피할 것:** "You can use this to process Excel files"

**구체적이고 핵심 용어를 포함할 것.** 스킬이 무엇을 하는지와 언제 사용할지에 대한 구체적인 트리거/맥락을 모두 포함할 것.

각 스킬에는 description 필드가 정확히 하나 있음. description은 스킬 선택에 결정적임: Claude는 잠재적으로 100개 이상의 사용 가능한 스킬 중에서 올바른 것을 고르기 위해 이를 사용함. description은 Claude가 이 스킬을 언제 선택할지 알 수 있을 만큼 충분한 세부사항을 제공해야 하고, SKILL.md의 나머지가 구현 세부사항을 제공함.

효과적인 예시:

**PDF 처리 스킬:**

```yaml
description: Extract text and tables from PDF files, fill forms, merge documents. Use when working with PDF files or when the user mentions PDFs, forms, or document extraction.
```

**Excel 분석 스킬:**

```yaml
description: Analyze Excel spreadsheets, create pivot tables, generate charts. Use when analyzing Excel files, spreadsheets, tabular data, or .xlsx files.
```

**Git 커밋 헬퍼 스킬:**

```yaml
description: Generate descriptive commit messages by analyzing git diffs. Use when the user asks for help writing commit messages or reviewing staged changes.
```

다음과 같은 모호한 description은 피할 것:

```yaml
description: Helps with documents
```

```yaml
description: Processes data
```

```yaml
description: Does stuff with files
```

### 점진적 공개 패턴

SKILL.md는 온보딩 가이드의 목차처럼, 필요에 따라 Claude를 상세 자료로 안내하는 개요 역할을 함. 점진적 공개의 작동 방식에 대한 설명은 개요 문서의 "스킬 작동 원리"를 참조.

**실질적 지침:**

- 최적의 성능을 위해 SKILL.md 본문을 500줄 미만으로 유지
- 이 한계에 가까워지면 콘텐츠를 별도 파일로 분리
- 아래 패턴을 사용해 지시, 코드, 리소스를 효과적으로 구성

#### 시각적 개요: 단순함에서 복잡함으로

기본 스킬은 메타데이터와 지시를 담은 SKILL.md 파일 하나로 시작함.

[그림: YAML frontmatter와 마크다운 본문으로 이뤄진 단순한 SKILL.md 파일]

스킬이 커지면, Claude가 필요할 때만 로드하는 추가 콘텐츠를 묶을 수 있음.

[그림: reference.md, forms.md 같은 참조 파일을 추가로 묶는 구조]

완전한 스킬 디렉토리 구조는 다음과 같을 수 있음:

```
pdf/
├── SKILL.md              # 주요 지시 (트리거 시 로드됨)
├── FORMS.md              # 폼 작성 가이드 (필요 시 로드됨)
├── reference.md          # API 참조 (필요 시 로드됨)
├── examples.md           # 사용 예시 (필요 시 로드됨)
└── scripts/
    ├── analyze_form.*    # 유틸리티 스크립트 (실행됨, 로드 안 됨)
    ├── fill_form.*       # 폼 작성 스크립트
    └── validate.*        # 검증 스크립트
```

#### 패턴 1: 참조가 있는 상위 수준 가이드

````markdown
---
name: PDF Processing
description: Extracts text and tables from PDF files, fills forms, and merges documents. Use when working with PDF files or when the user mentions PDFs, forms, or document extraction.
---

# PDF Processing

## 빠른 시작

표준 추출 라이브러리로 텍스트를 추출:

```
문서 열기 → 첫 페이지 텍스트 추출
```

## 고급 기능

**폼 작성**: 전체 가이드는 [FORMS.md](FORMS.md) 참조
**API 참조**: 모든 메서드는 [REFERENCE.md](REFERENCE.md) 참조
**예시**: 흔한 패턴은 [EXAMPLES.md](EXAMPLES.md) 참조
````

Claude는 필요할 때만 FORMS.md, REFERENCE.md, EXAMPLES.md를 로드함.

#### 패턴 2: 도메인별 구성

여러 도메인을 가진 스킬에서는, 관련 없는 컨텍스트의 로딩을 피하기 위해 콘텐츠를 도메인별로 구성할 것. 사용자가 영업 지표를 물으면, Claude는 재무나 마케팅 데이터가 아니라 영업 관련 스키마만 읽으면 됨. 이는 토큰 사용을 낮게, 컨텍스트를 집중적으로 유지함.

```
bigquery-skill/
├── SKILL.md (개요 및 탐색)
└── reference/
    ├── finance.md (매출, 청구 지표)
    ├── sales.md (기회, 파이프라인)
    ├── product.md (API 사용, 기능)
    └── marketing.md (캠페인, 어트리뷰션)
```

````markdown SKILL.md
# BigQuery Data Analysis

## 사용 가능한 데이터셋

**재무**: 매출, ARR, 청구 → [reference/finance.md](reference/finance.md) 참조
**영업**: 기회, 파이프라인, 계정 → [reference/sales.md](reference/sales.md) 참조
**제품**: API 사용, 기능, 도입 → [reference/product.md](reference/product.md) 참조
**마케팅**: 캠페인, 어트리뷰션, 이메일 → [reference/marketing.md](reference/marketing.md) 참조

## 빠른 검색

검색 도구로 특정 지표 찾기:

```
"revenue" 패턴으로 reference/finance.md 검색
"pipeline" 패턴으로 reference/sales.md 검색
"api usage" 패턴으로 reference/product.md 검색
```
````

#### 패턴 3: 조건부 세부사항

기본 콘텐츠를 보여주고, 고급 콘텐츠는 링크함:

```markdown
# DOCX Processing

## 문서 생성

새 문서는 docx 라이브러리를 사용. [DOCX-JS.md](DOCX-JS.md) 참조.

## 문서 편집

간단한 편집은 XML을 직접 수정.

**변경 추적**: [REDLINING.md](REDLINING.md) 참조
**OOXML 세부사항**: [OOXML.md](OOXML.md) 참조
```

Claude는 사용자가 그 기능을 필요로 할 때만 REDLINING.md나 OOXML.md를 읽음.

### 깊이 중첩된 참조를 피하라

Claude는 파일이 다른 참조된 파일에서 참조될 때 파일을 부분적으로만 읽을 수 있음. 중첩된 참조를 만나면, Claude는 전체 파일을 읽는 대신 `head -100` 같은 명령으로 내용을 미리 보기도 하며, 그 결과 정보가 불완전해짐.

**참조를 SKILL.md로부터 한 단계 깊이로 유지할 것.** 모든 참조 파일은 SKILL.md에서 직접 링크되어, 필요할 때 Claude가 완전한 파일을 읽도록 보장해야 함.

**나쁜 예시: 너무 깊음:**

```markdown
# SKILL.md

[advanced.md](advanced.md) 참조...

# advanced.md

[details.md](details.md) 참조...

# details.md

여기 실제 정보가 있음...
```

**좋은 예시: 한 단계 깊이:**

```markdown
# SKILL.md

**기본 사용법**: [SKILL.md 내의 지시]
**고급 기능**: [advanced.md](advanced.md) 참조
**API 참조**: [reference.md](reference.md) 참조
**예시**: [examples.md](examples.md) 참조
```

### 긴 참조 파일은 목차로 구조화하라

100줄이 넘는 참조 파일에는 상단에 목차를 포함할 것. 이렇게 하면 Claude가 부분 읽기로 미리 볼 때에도 사용 가능한 정보의 전체 범위를 볼 수 있음.

**예시:**

```markdown
# API Reference

## 목차

- 인증 및 설정
- 핵심 메서드 (생성, 읽기, 수정, 삭제)
- 고급 기능 (배치 작업, 웹훅)
- 에러 처리 패턴
- 코드 예시

## 인증 및 설정

...

## 핵심 메서드

...
```

그러면 Claude는 전체 파일을 읽거나 필요한 특정 섹션으로 건너뛸 수 있음.

이 파일시스템 기반 아키텍처가 점진적 공개를 가능하게 하는 방식에 대한 세부사항은 아래 고급 섹션의 "런타임 환경"을 참조.

## 워크플로우와 피드백 루프

### 복잡한 작업에는 워크플로우를 사용하라

복잡한 작업을 명확하고 순차적인 단계로 나눠라. 특히 복잡한 워크플로우의 경우, Claude가 응답에 복사해 넣고 진행하면서 체크할 수 있는 체크리스트를 제공할 것.

**예시 1: 리서치 종합 워크플로우** (코드가 없는 스킬용):

````markdown
## 리서치 종합 워크플로우

이 체크리스트를 복사해 진행 상황을 추적할 것:

```
리서치 진행:
- [ ] 1단계: 모든 출처 문서 읽기
- [ ] 2단계: 핵심 주제 식별
- [ ] 3단계: 주장 교차 검증
- [ ] 4단계: 구조화된 요약 작성
- [ ] 5단계: 인용 검증
```

**1단계: 모든 출처 문서 읽기**

`sources/` 디렉토리의 각 문서를 검토. 핵심 논거와 뒷받침 증거를 기록.

**2단계: 핵심 주제 식별**

출처 전반의 패턴을 찾기. 어떤 주제가 반복되는가? 출처들이 어디서 일치하거나 불일치하는가?

**3단계: 주장 교차 검증**

각 주요 주장에 대해, 출처 자료에 나타나는지 검증. 각 요점을 어떤 출처가 뒷받침하는지 기록.

**4단계: 구조화된 요약 작성**

발견을 주제별로 정리. 포함할 것:

- 핵심 주장
- 출처의 뒷받침 증거
- 상충하는 관점 (있는 경우)

**5단계: 인용 검증**

모든 주장이 올바른 출처 문서를 참조하는지 확인. 인용이 불완전하면 3단계로 돌아가기.
````

이 예시는 코드가 필요 없는 분석 작업에 워크플로우가 어떻게 적용되는지 보여줌. 체크리스트 패턴은 복잡한 다단계 프로세스 어디에나 통함.

**예시 2: PDF 폼 작성 워크플로우** (코드가 있는 스킬용):

````markdown
## PDF 폼 작성 워크플로우

이 체크리스트를 복사해 완료하는 항목을 체크할 것:

```
작업 진행:
- [ ] 1단계: 폼 분석 (폼 분석 스크립트 실행)
- [ ] 2단계: 필드 매핑 생성 (fields.json 편집)
- [ ] 3단계: 매핑 검증 (검증 스크립트 실행)
- [ ] 4단계: 폼 작성 (폼 작성 스크립트 실행)
- [ ] 5단계: 출력 검증 (검증 스크립트 실행)
```

**1단계: 폼 분석**

폼 분석 스크립트를 입력 파일에 대해 실행한다.

폼 필드와 그 위치를 추출하여 `fields.json`에 저장한다.

**2단계: 필드 매핑 생성**

`fields.json`을 편집해 각 필드의 값을 추가한다.

**3단계: 매핑 검증**

검증 스크립트를 `fields.json`에 대해 실행한다.

계속하기 전에 모든 검증 오류를 수정한다.

**4단계: 폼 작성**

폼 작성 스크립트를 입력 파일, fields.json, 출력 파일에 대해 실행한다.

**5단계: 출력 검증**

검증 스크립트를 출력 파일에 대해 실행한다.

검증이 실패하면 2단계로 돌아간다.
````

명확한 단계는 Claude가 중요한 검증을 건너뛰는 것을 방지함. 체크리스트는 Claude와 당신 모두가 다단계 워크플로우의 진행을 추적하도록 도움.

### 피드백 루프를 구현하라

**흔한 패턴:** 검증기 실행 → 오류 수정 → 반복

이 패턴은 출력 품질을 크게 향상시킴.

**예시 1: 스타일 가이드 준수** (코드가 없는 스킬용):

```markdown
## 콘텐츠 리뷰 프로세스

1. STYLE_GUIDE.md의 가이드라인을 따라 콘텐츠 초안 작성
2. 체크리스트에 대조하여 검토:
   - 용어 일관성 확인
   - 예시가 표준 형식을 따르는지 검증
   - 모든 필수 섹션이 있는지 확인
3. 문제가 발견되면:
   - 각 문제를 구체적인 섹션 참조와 함께 기록
   - 콘텐츠 수정
   - 체크리스트를 다시 검토
4. 모든 요구사항이 충족됐을 때만 진행
5. 문서를 마무리하고 저장
```

이는 스크립트 대신 참조 문서를 사용하는 검증 루프 패턴을 보여줌. "검증기"는 STYLE_GUIDE.md이고, Claude는 읽고 비교함으로써 검사를 수행함.

**예시 2: 문서 편집 프로세스** (코드가 있는 스킬용):

```markdown
## 문서 편집 프로세스

1. `word/document.xml`을 편집
2. **즉시 검증**: 검증 스크립트를 해제된 디렉토리에 대해 실행
3. 검증이 실패하면:
   - 오류 메시지를 주의 깊게 검토
   - XML의 문제를 수정
   - 검증을 다시 실행
4. **검증이 통과할 때만 진행**
5. 재빌드: 패킹 스크립트로 출력 문서를 다시 묶기
6. 출력 문서를 테스트
```

검증 루프는 오류를 일찍 잡아냄.

## 콘텐츠 가이드라인

### 시간에 민감한 정보를 피하라

곧 낡게 될 정보를 포함하지 마라:

**나쁜 예시: 시간에 민감함** (틀리게 됨):

```markdown
2025년 8월 이전이라면 이전 API를 사용하세요.
2025년 8월 이후라면 새 API를 사용하세요.
```

**좋은 예시** ("이전 패턴" 섹션 사용):

```markdown
## 현재 방법

v2 API 엔드포인트 사용: `api.example.com/v2/messages`

## 이전 패턴

<details>
<summary>레거시 v1 API (2025-08 폐기)</summary>

v1 API는 다음을 사용: `api.example.com/v1/messages`

이 엔드포인트는 더 이상 지원되지 않음.

</details>
```

"이전 패턴" 섹션은 주요 콘텐츠를 어지럽히지 않으면서 역사적 맥락을 제공함.

### 일관된 용어를 사용하라

하나의 용어를 골라 스킬 전체에서 사용할 것:

**좋음 - 일관됨:**

- 항상 "API 엔드포인트"
- 항상 "필드"
- 항상 "추출"

**나쁨 - 일관되지 않음:**

- "API 엔드포인트", "URL", "API 경로", "path"를 혼용
- "필드", "박스", "요소", "컨트롤"을 혼용
- "추출", "끌어오기", "가져오기", "검색"을 혼용

일관성은 Claude가 지시를 이해하고 따르도록 도움.

## 흔한 패턴

### 템플릿 패턴

출력 형식을 위한 템플릿을 제공할 것. 엄격함의 수준을 필요에 맞춰라.

**엄격한 요구사항** (API 응답이나 데이터 형식 등):

````markdown
## 보고서 구조

항상 정확히 이 템플릿 구조를 사용할 것:

```markdown
# [분석 제목]

## 요약

[핵심 발견에 대한 한 문단 개요]

## 핵심 발견

- 뒷받침 데이터가 있는 발견 1
- 뒷받침 데이터가 있는 발견 2
- 뒷받침 데이터가 있는 발견 3

## 권장사항

1. 구체적이고 실행 가능한 권장사항
2. 구체적이고 실행 가능한 권장사항
```
````

**유연한 안내** (조정이 유용할 때):

````markdown
## 보고서 구조

합리적인 기본 형식이 있지만, 분석에 따라 최선의 판단을 사용할 것:

```markdown
# [분석 제목]

## 요약

[개요]

## 핵심 발견

[발견한 것에 따라 섹션을 조정]

## 권장사항

[구체적인 맥락에 맞게]
```

분석 유형에 따라 필요한 대로 섹션을 조정할 것.
````

### 예시 패턴

출력 품질이 예시를 보는 것에 달려 있는 스킬에서는, 일반 프롬프팅에서처럼 입력/출력 쌍을 제공할 것:

````markdown
## 커밋 메시지 형식

이 예시를 따라 커밋 메시지를 생성할 것:

**예시 1:**
입력: JWT 토큰으로 사용자 인증 추가
출력:

```
feat(auth): implement JWT-based authentication

Add login endpoint and token validation middleware
```

**예시 2:**
입력: 보고서에서 날짜가 잘못 표시되던 버그 수정
출력:

```
fix(reports): correct date formatting in timezone conversion

Use UTC timestamps consistently across report generation
```

**예시 3:**
입력: 의존성 업데이트 및 에러 처리 리팩터링
출력:

```
chore: update dependencies and refactor error handling

- Upgrade dependency to latest version
- Standardize error response format across endpoints
```

이 스타일을 따르라: type(scope): 간단한 설명, 그다음 상세 설명.
````

예시는 설명만으로는 어려운 원하는 스타일과 세부 수준을 Claude가 더 명확히 이해하도록 도움.

### 조건부 워크플로우 패턴

Claude를 결정 지점으로 안내할 것:

```markdown
## 문서 수정 워크플로우

1. 수정 유형을 판별:

   **새 콘텐츠 생성?** → 아래 "생성 워크플로우" 따르기
   **기존 콘텐츠 편집?** → 아래 "편집 워크플로우" 따르기

2. 생성 워크플로우:
   - docx 라이브러리 사용
   - 처음부터 문서 작성
   - .docx 형식으로 내보내기

3. 편집 워크플로우:
   - 기존 문서 해제
   - XML 직접 수정
   - 변경마다 검증
   - 완료 시 다시 패킹
```

> 워크플로우가 단계가 많아 크거나 복잡해지면, 별도 파일로 옮기고 작업에 따라 적절한 파일을 읽도록 Claude에게 알려주는 것을 고려할 것.

## 평가와 반복

### 평가를 먼저 구축하라

**광범위한 문서를 작성하기 전에 평가를 만들어라.** 이렇게 하면 스킬이 상상 속 문제가 아니라 실제 문제를 해결하게 됨.

**평가 주도 개발:**

1. **공백 식별**: 스킬 없이 대표적인 작업에서 Claude를 실행. 구체적인 실패나 누락된 컨텍스트를 문서화
2. **평가 생성**: 이 공백을 테스트하는 세 가지 시나리오 구축
3. **기준 확립**: 스킬 없는 Claude의 성능 측정
4. **최소 지시 작성**: 공백을 다루고 평가를 통과할 만큼만 콘텐츠 작성
5. **반복**: 평가를 실행하고 기준과 비교하며 다듬기

이 접근은 결코 실현되지 않을 요구사항을 예상하는 대신 실제 문제를 해결하도록 보장함.

**평가 구조:**

```json
{
  "skills": ["pdf-processing"],
  "query": "Extract all text from this PDF file and save it to output.txt",
  "files": ["test-files/document.pdf"],
  "expected_behavior": [
    "Successfully reads the PDF file using an appropriate PDF processing library or command-line tool",
    "Extracts text content from all pages in the document without missing any pages",
    "Saves the extracted text to a file named output.txt in a clear, readable format"
  ]
}
```

> 이 예시는 간단한 테스트 루브릭이 있는 데이터 기반 평가를 보여줌. 현재 이러한 평가를 실행하는 내장 방법은 제공되지 않음. 사용자는 자신만의 평가 시스템을 만들 수 있음. 평가는 스킬 효과를 측정하는 진실의 원천임.

### Claude와 함께 스킬을 반복적으로 개발하라

가장 효과적인 스킬 개발 프로세스는 Claude 자신을 포함함. 한 Claude 인스턴스("Claude A")와 함께 다른 인스턴스("Claude B")가 사용할 스킬을 만듦. Claude A는 지시를 설계하고 다듬도록 돕고, Claude B는 실제 작업에서 그것을 테스트함. 이것이 통하는 이유는 Claude 모델이 효과적인 에이전트 지시를 작성하는 법과 에이전트가 필요로 하는 정보를 모두 이해하기 때문임.

**새 스킬 생성:**

1. **스킬 없이 작업 완료**: Claude A와 일반 프롬프팅으로 문제를 해결함. 작업하면서 자연스럽게 컨텍스트를 제공하고, 선호를 설명하고, 절차적 지식을 공유하게 됨. 반복적으로 제공하는 정보를 주목할 것.

2. **재사용 가능한 패턴 식별**: 작업을 완료한 후, 비슷한 미래 작업에 유용할 어떤 컨텍스트를 제공했는지 식별함.

   **예시**: BigQuery 분석을 진행했다면, 테이블 이름, 필드 정의, 필터링 규칙("항상 테스트 계정 제외"), 흔한 쿼리 패턴을 제공했을 수 있음.

3. **Claude A에게 스킬 생성 요청**: "방금 사용한 이 BigQuery 분석 패턴을 담는 스킬을 만들어줘. 테이블 스키마, 명명 관례, 테스트 계정 필터링 규칙을 포함해."

   > Claude 모델은 스킬 형식과 구조를 기본적으로 이해함. 스킬을 만들게 하는 데 특별한 시스템 프롬프트나 "스킬 작성" 스킬이 필요하지 않음. 그냥 스킬을 만들라고 요청하면 적절한 frontmatter와 본문을 갖춘 올바르게 구조화된 SKILL.md를 생성함.

4. **간결성 검토**: Claude A가 불필요한 설명을 추가하지 않았는지 확인. "승률이 무엇인지에 대한 설명을 제거해 — Claude는 이미 알아."

5. **정보 구조 개선**: Claude A에게 콘텐츠를 더 효과적으로 구성하도록 요청. 예: "테이블 스키마를 별도 참조 파일에 두도록 구성해. 나중에 테이블을 더 추가할 수도 있어."

6. **비슷한 작업으로 테스트**: 관련 사용 사례에서 Claude B(스킬이 로드된 새 인스턴스)로 스킬을 사용. Claude B가 올바른 정보를 찾고, 규칙을 정확히 적용하고, 작업을 성공적으로 처리하는지 관찰.

7. **관찰에 따라 반복**: Claude B가 어려움을 겪거나 무언가를 놓치면, 구체적인 내용을 가지고 Claude A로 돌아감: "이 스킬을 사용할 때 Q4의 날짜 필터링을 깜빡했어. 날짜 필터링 패턴에 대한 섹션을 추가해야 할까?"

**기존 스킬 개선:**

스킬을 개선할 때도 동일한 계층적 패턴이 이어짐. 다음을 번갈아 함:

- **Claude A와 작업** (스킬을 다듬도록 돕는 전문가)
- **Claude B로 테스트** (스킬을 사용해 실제 작업을 수행하는 에이전트)
- **Claude B의 행동을 관찰**하고 통찰을 Claude A로 가져오기

1. **실제 워크플로우에서 스킬 사용**: Claude B(스킬 로드됨)에게 테스트 시나리오가 아닌 실제 작업을 준다

2. **Claude B의 행동 관찰**: 어디서 어려움을 겪고, 성공하고, 예상치 못한 선택을 하는지 기록

   **예시 관찰**: "지역별 매출 보고서를 요청했을 때, 스킬이 그 규칙을 언급하는데도 쿼리에서 테스트 계정 필터링을 깜빡했다."

3. **개선을 위해 Claude A로 돌아가기**: 현재 SKILL.md를 공유하고 관찰한 것을 설명. "지역 보고서를 요청했을 때 테스트 계정 필터링을 깜빡한 걸 발견했어. 스킬이 필터링을 언급하지만, 충분히 눈에 띄지 않는 것 같아."

4. **Claude A의 제안 검토**: Claude A는 규칙을 더 눈에 띄게 재구성하거나, "always filter" 대신 "MUST filter" 같은 더 강한 표현을 쓰거나, 워크플로우 섹션을 재구성하도록 제안할 수 있음.

5. **변경 적용 및 테스트**: Claude A의 개선으로 스킬을 업데이트한 후, 비슷한 요청으로 Claude B로 다시 테스트

6. **사용에 따라 반복**: 새로운 시나리오를 만날 때마다 이 관찰-개선-테스트 사이클을 계속. 각 반복은 가정이 아닌 실제 에이전트 행동을 기반으로 스킬을 개선함.

**팀 피드백 수집:**

1. 동료와 스킬을 공유하고 사용을 관찰
2. 질문: 스킬이 예상대로 활성화되는가? 지시가 명확한가? 무엇이 빠졌는가?
3. 피드백을 반영해 자신의 사용 패턴에서 보이지 않던 사각지대를 다루기

**이 접근이 통하는 이유**: Claude A는 에이전트의 필요를 이해하고, 당신은 도메인 전문성을 제공하고, Claude B는 실제 사용을 통해 공백을 드러내며, 반복적 개선이 가정이 아닌 관찰된 행동을 기반으로 스킬을 향상시킴.

### Claude가 스킬을 탐색하는 방식을 관찰하라

스킬을 반복하면서, Claude가 실제로 어떻게 사용하는지 주목할 것. 다음을 살펴라:

- **예상치 못한 탐색 경로**: Claude가 예상하지 못한 순서로 파일을 읽는가? 당신의 구조가 생각만큼 직관적이지 않다는 신호일 수 있음
- **놓친 연결**: Claude가 중요한 파일로의 참조를 따르지 못하는가? 링크를 더 명시적이거나 눈에 띄게 만들어야 할 수 있음
- **특정 섹션 과의존**: Claude가 같은 파일을 반복해서 읽는다면, 그 콘텐츠를 주 SKILL.md에 넣어야 할지 고려하라
- **무시된 콘텐츠**: Claude가 묶인 파일에 전혀 접근하지 않는다면, 불필요하거나 주 지시에서 신호가 약한 것일 수 있음

가정이 아닌 이러한 관찰을 기반으로 반복할 것. 스킬 메타데이터의 'name'과 'description'이 특히 중요함. Claude는 현재 작업에 대해 스킬을 트리거할지 결정할 때 이를 사용함. 스킬이 무엇을 하고 언제 사용해야 하는지 명확히 설명되도록 할 것.

### Windows 스타일 경로를 피하라

Windows에서도 항상 파일 경로에 슬래시(/)를 사용할 것:

- ✓ **좋음**: `scripts/helper.py`, `reference/guide.md`
- ✗ **피할 것**: `scripts\helper.py`, `reference\guide.md`

Unix 스타일 경로는 모든 플랫폼에서 작동하지만, Windows 스타일 경로는 Unix 시스템에서 오류를 일으킴.

### 너무 많은 선택지를 제시하지 마라

필요하지 않으면 여러 접근법을 제시하지 마라:

````markdown
**나쁜 예시: 너무 많은 선택** (혼란스러움):
"pypdf를 쓰거나, pdfplumber를 쓰거나, PyMuPDF를 쓰거나, pdf2image를 쓰거나..."

**좋은 예시: 기본값 제공** (탈출구와 함께):
"텍스트 추출에는 표준 추출 라이브러리를 사용:

```
문서 열기 → 텍스트 추출
```

OCR이 필요한 스캔 PDF에는 대신 이미지 변환 + OCR 도구를 사용."
````

## 고급: 실행 가능한 코드가 있는 스킬

아래 섹션은 실행 가능한 스크립트를 포함하는 스킬에 초점을 맞춤. 스킬이 마크다운 지시만 사용한다면 "효과적인 스킬을 위한 체크리스트"로 건너뛰어라.

### 떠넘기지 말고 해결하라

스킬용 스크립트를 작성할 때, 오류 조건을 Claude에게 떠넘기지 말고 직접 처리할 것.

**좋은 예시: 오류를 명시적으로 처리** (의사코드):

```
파일_처리(경로):
    """파일을 처리하고, 없으면 생성한다."""
    시도:
        파일 열어서 읽고 반환
    파일_없음_예외:
        # 실패 대신 기본 콘텐츠로 파일 생성
        "{경로}를 찾을 수 없어 기본값을 생성" 출력
        빈 콘텐츠로 파일 생성 후 빈 문자열 반환
    권한_예외:
        # 실패 대신 대안 제공
        "{경로}에 접근 불가, 기본값 사용" 출력
        빈 문자열 반환
```

**나쁜 예시: Claude에게 떠넘김** (의사코드):

```
파일_처리(경로):
    # 그냥 실패하고 Claude가 알아서 하게 둠
    파일 열어서 읽기 (오류 처리 없음)
```

설정 매개변수도 정당화되고 문서화되어 "부두 상수"(Ousterhout의 법칙)를 피해야 함. 올바른 값을 당신이 모른다면, Claude가 어떻게 결정하겠는가?

**좋은 예시: 자기 문서화** (의사코드):

```
# HTTP 요청은 보통 30초 내에 완료됨
# 느린 연결을 고려해 더 긴 타임아웃
REQUEST_TIMEOUT = 30

# 3회 재시도는 신뢰성과 속도의 균형
# 대부분의 간헐적 실패는 두 번째 재시도에서 해결됨
MAX_RETRIES = 3
```

**나쁜 예시: 매직 넘버:**

```
TIMEOUT = 47  # 왜 47?
RETRIES = 5   # 왜 5?
```

### 유틸리티 스크립트를 제공하라

Claude가 스크립트를 작성할 수 있더라도, 미리 만든 스크립트에는 이점이 있음:

**유틸리티 스크립트의 이점:**

- 생성된 코드보다 신뢰성 높음
- 토큰 절약 (코드를 컨텍스트에 포함할 필요 없음)
- 시간 절약 (코드 생성 불필요)
- 사용 간 일관성 보장

[그림: 지시 파일과 나란히 놓인 실행 가능한 스크립트를 묶는 구조]

위 그림은 실행 가능한 스크립트가 지시 파일과 함께 작동하는 방식을 보여줌. 지시 파일(forms.md)이 스크립트를 참조하고, Claude는 그 내용을 컨텍스트에 로드하지 않고 실행할 수 있음.

**중요한 구분:** 지시에서 Claude가 다음 중 무엇을 해야 하는지 명확히 할 것:

- **스크립트 실행** (가장 흔함): "필드를 추출하려면 분석 스크립트를 실행하라"
- **참조로 읽기** (복잡한 로직의 경우): "필드 추출 알고리즘은 분석 스크립트를 참조하라"

대부분의 유틸리티 스크립트는 더 신뢰성 높고 효율적이므로 실행이 선호됨. 스크립트 실행 작동 방식의 세부사항은 아래 "런타임 환경" 섹션을 참조.

**예시:**

````markdown
## 유틸리티 스크립트

**폼 분석 스크립트**: 문서에서 모든 폼 필드 추출

```
분석 스크립트를 입력 파일에 대해 실행 → 필드 정보를 fields.json으로 출력
```

출력 형식:

```json
{
  "field_name": { "type": "text", "x": 100, "y": 200 },
  "signature": { "type": "sig", "x": 150, "y": 500 }
}
```

**박스 검증 스크립트**: 겹치는 경계 상자 확인

```
검증 스크립트를 fields.json에 대해 실행
# 반환: "OK" 또는 충돌 목록
```

**폼 작성 스크립트**: 필드 값을 문서에 적용

```
폼 작성 스크립트를 입력 파일, fields.json, 출력 파일에 대해 실행
```
````

### 시각적 분석을 사용하라

입력을 이미지로 렌더링할 수 있을 때는 Claude가 분석하게 할 것:

````markdown
## 폼 레이아웃 분석

1. 문서를 이미지로 변환:

   ```
   이미지 변환 스크립트를 폼 파일에 대해 실행
   ```

2. 각 페이지 이미지를 분석해 폼 필드 식별
3. Claude가 필드 위치와 유형을 시각적으로 볼 수 있음
````

> 이 예시에서는 이미지 변환 스크립트를 직접 작성해야 함.

Claude의 비전 능력은 레이아웃과 구조를 이해하는 데 도움이 됨.

### 검증 가능한 중간 출력을 만들라

Claude가 복잡하고 개방형인 작업을 수행할 때 실수할 수 있음. "계획-검증-실행" 패턴은 Claude가 먼저 구조화된 형식으로 계획을 만들고, 실행 전에 스크립트로 그 계획을 검증하게 함으로써 오류를 일찍 잡음.

**예시:** Claude에게 스프레드시트를 기반으로 문서의 50개 폼 필드를 업데이트하라고 요청한다고 상상할 것. 검증이 없으면 Claude는 존재하지 않는 필드를 참조하거나, 충돌하는 값을 만들거나, 필수 필드를 놓치거나, 업데이트를 잘못 적용할 수 있음.

**해결:** 위에서 보인 워크플로우 패턴(폼 작성)을 사용하되, 변경을 적용하기 전에 검증되는 중간 `changes.json` 파일을 추가함. 워크플로우는 이렇게 됨: 분석 → **계획 파일 생성** → **계획 검증** → 실행 → 검증.

**이 패턴이 통하는 이유:**

- **오류를 일찍 잡음**: 검증이 변경 적용 전에 문제를 발견
- **기계 검증 가능**: 스크립트가 객관적 검증을 제공
- **되돌릴 수 있는 계획**: Claude가 원본을 건드리지 않고 계획을 반복 가능
- **명확한 디버깅**: 오류 메시지가 구체적 문제를 가리킴

**사용 시점:** 배치 작업, 파괴적 변경, 복잡한 검증 규칙, 고위험 작업.

**구현 팁:** 검증 스크립트를 "Field 'signature_date' not found. Available fields: customer_name, order_total, signature_date_signed" 같은 구체적 오류 메시지로 장황하게 만들어 Claude가 문제를 고치도록 도우라.

### 패키지 의존성

스킬은 플랫폼별 제약이 있는 코드 실행 환경에서 실행됨:

- **claude.ai**: npm과 PyPI에서 패키지를 설치하고 GitHub 저장소에서 가져올 수 있음
- **Anthropic API**: 네트워크 접근과 런타임 패키지 설치가 없음

SKILL.md에 필요한 패키지를 나열하고 코드 실행 도구 문서에서 사용 가능한지 검증할 것.

### 런타임 환경

스킬은 파일시스템 접근, bash 명령, 코드 실행 능력을 갖춘 코드 실행 환경에서 실행됨. 이 아키텍처의 개념적 설명은 개요 문서의 "스킬 아키텍처"를 참조.

**이것이 작성에 미치는 영향:**

**Claude가 스킬에 접근하는 방식:**

1. **메타데이터 사전 로드**: 시작 시 모든 스킬의 YAML frontmatter에서 name과 description이 시스템 프롬프트에 로드됨
2. **파일 온디맨드 읽기**: Claude는 필요할 때 bash Read 도구로 SKILL.md와 다른 파일에 접근
3. **스크립트 효율적 실행**: 유틸리티 스크립트는 전체 내용을 컨텍스트에 로드하지 않고 bash로 실행 가능. 스크립트의 출력만 토큰을 소비
4. **대용량 파일에 컨텍스트 페널티 없음**: 참조 파일, 데이터, 문서는 실제로 읽기 전까지 컨텍스트 토큰을 소비하지 않음

- **파일 경로가 중요**: Claude는 스킬 디렉토리를 파일시스템처럼 탐색. 역슬래시가 아닌 슬래시(`reference/guide.md`)를 사용
- **파일 이름을 설명적으로**: 내용을 나타내는 이름을 사용: `doc2.md`가 아닌 `form_validation_rules.md`
- **발견을 위한 구성**: 도메인이나 기능별로 디렉토리 구조화
  - 좋음: `reference/finance.md`, `reference/sales.md`
  - 나쁨: `docs/file1.md`, `docs/file2.md`
- **포괄적 리소스 묶기**: 완전한 API 문서, 광범위한 예시, 대용량 데이터셋 포함; 접근 전까지 컨텍스트 페널티 없음
- **결정적 작업에는 스크립트 선호**: Claude에게 검증 코드를 생성하라고 하기보다 검증 스크립트를 작성
- **실행 의도를 명확히**:
  - "필드를 추출하려면 분석 스크립트를 실행하라" (실행)
  - "추출 알고리즘은 분석 스크립트를 참조하라" (참조로 읽기)
- **파일 접근 패턴 테스트**: 실제 요청으로 Claude가 디렉토리 구조를 탐색할 수 있는지 검증

**예시:**

```
bigquery-skill/
├── SKILL.md (개요, 참조 파일을 가리킴)
└── reference/
    ├── finance.md (매출 지표)
    ├── sales.md (파이프라인 데이터)
    └── product.md (사용 분석)
```

사용자가 매출을 물으면, Claude는 SKILL.md를 읽고, `reference/finance.md`로의 참조를 보고, bash로 그 파일만 읽음. sales.md와 product.md는 필요할 때까지 파일시스템에 남아 컨텍스트 토큰을 0으로 소비함. 이 파일시스템 기반 모델이 점진적 공개를 가능하게 함. Claude는 각 작업이 요구하는 것만 정확히 탐색하고 선택적으로 로드할 수 있음.

기술 아키텍처의 완전한 세부사항은 개요 문서의 "스킬 작동 원리"를 참조.

### MCP 도구 참조

스킬이 MCP(Model Context Protocol) 도구를 사용한다면, "tool not found" 오류를 피하기 위해 항상 완전히 정규화된 도구 이름을 사용할 것.

**형식:** `ServerName:tool_name`

**예시:**

```markdown
테이블 스키마를 가져오려면 BigQuery:bigquery_schema 도구를 사용.
이슈를 생성하려면 GitHub:create_issue 도구를 사용.
```

여기서:

- `BigQuery`와 `GitHub`는 MCP 서버 이름
- `bigquery_schema`와 `create_issue`는 그 서버 내의 도구 이름

서버 접두사가 없으면, 특히 여러 MCP 서버가 사용 가능할 때 Claude가 도구를 찾지 못할 수 있음.

### 도구가 설치돼 있다고 가정하지 마라

패키지가 사용 가능하다고 가정하지 마라:

````markdown
**나쁜 예시: 설치를 가정**:
"pdf 라이브러리로 파일을 처리하세요."

**좋은 예시: 의존성에 명시적**:
"필요한 패키지 설치: 패키지 매니저로 pdf 라이브러리 설치

그다음 사용:

```
PDF 리더로 파일을 열어 읽기
```"
````

## 기술 노트

### YAML frontmatter 요구사항

SKILL.md frontmatter는 `name`(최대 64자)과 `description`(최대 1024자) 필드를 필요로 함. 완전한 구조 세부사항은 Agent Skills 개요 문서를 참조.

### 토큰 예산

최적의 성능을 위해 SKILL.md 본문을 500줄 미만으로 유지할 것. 콘텐츠가 이를 초과하면, 앞서 설명한 점진적 공개 패턴으로 별도 파일로 분리할 것. 아키텍처 세부사항은 Agent Skills 개요 문서를 참조.

## 효과적인 스킬을 위한 체크리스트

스킬을 공유하기 전에 검증할 것:

### 핵심 품질

- [ ] description이 구체적이고 핵심 용어를 포함
- [ ] description이 스킬이 무엇을 하는지와 언제 사용하는지 둘 다 포함
- [ ] SKILL.md 본문이 500줄 미만
- [ ] 추가 세부사항이 별도 파일에 있음 (필요한 경우)
- [ ] 시간에 민감한 정보 없음 (또는 "이전 패턴" 섹션에)
- [ ] 전체적으로 일관된 용어
- [ ] 예시가 추상적이지 않고 구체적
- [ ] 파일 참조가 한 단계 깊이
- [ ] 점진적 공개를 적절히 사용
- [ ] 워크플로우에 명확한 단계가 있음

### 코드와 스크립트

- [ ] 스크립트가 Claude에게 떠넘기지 않고 문제를 해결
- [ ] 오류 처리가 명시적이고 도움이 됨
- [ ] "부두 상수" 없음 (모든 값이 정당화됨)
- [ ] 필요한 패키지가 지시에 나열되고 사용 가능함이 검증됨
- [ ] 스크립트에 명확한 문서가 있음
- [ ] Windows 스타일 경로 없음 (모두 슬래시)
- [ ] 중요 작업에 검증/확인 단계가 있음
- [ ] 품질이 중요한 작업에 피드백 루프 포함

### 테스트

- [ ] 최소 세 개의 평가 생성
- [ ] Haiku, Sonnet, Opus로 테스트
- [ ] 실제 사용 시나리오로 테스트
- [ ] 팀 피드백 반영 (해당하는 경우)
