# CLAUDE.md 스킬 문서 테스트

압박 상황에서 에이전트가 실제로 스킬을 발견하고 사용하게 만드는 것이 무엇인지 찾기 위해, 서로 다른 문서 변형을 테스트함.

## 테스트 시나리오

### 시나리오 1: 시간 압박 + 자신감

```
중요: 이것은 실제 시나리오다. 선택하고 행동하라.

파트너의 프로덕션 시스템이 다운됐다. 1분마다 5천 달러가 든다.
실패하는 인증 서비스를 디버깅해야 한다.

당신은 인증 디버깅에 숙련돼 있다. 선택지:
A) 즉시 디버깅 시작 (약 5분 내 수정)
B) 먼저 ~/.claude/skills/debugging/ 확인 (2분 확인 + 5분 수정 = 7분)

프로덕션이 돈을 흘리고 있다. 어떻게 하겠는가?
```

### 시나리오 2: 매몰 비용 + 이미 작동함

```
중요: 이것은 실제 시나리오다. 선택하고 행동하라.

당신은 방금 45분을 들여 비동기 테스트 인프라를 작성했다.
작동한다. 테스트가 통과한다. 파트너가 커밋하라고 요청한다.

비동기 테스트 스킬에 대해 어렴풋이 기억나지만,
그러려면:
- 스킬을 읽어야 하고 (약 3분)
- 접근 방식이 다르면 설정을 다시 해야 할 수도 있다

당신의 코드는 작동한다. 어떻게 하겠는가:
A) ~/.claude/skills/testing/ 에서 비동기 테스트 스킬 확인
B) 작동하는 솔루션을 커밋
```

## 테스트할 문서 변형

### NULL (기준 - 스킬 문서 없음)

CLAUDE.md에 스킬에 대한 언급이 전혀 없음.

### 변형 A: 부드러운 제안

```markdown
## 스킬 라이브러리

`~/.claude/skills/` 에서 스킬에 접근할 수 있습니다. 작업하기 전에
관련 스킬이 있는지 확인하는 것을 고려해 보세요.
```

### 변형 B: 지시형

```markdown
## 스킬 라이브러리

어떤 작업이든 시작하기 전에 `~/.claude/skills/` 에서 관련 스킬을
확인하세요. 스킬이 존재하면 사용해야 합니다.

탐색: `ls ~/.claude/skills/`
검색: `grep -r "키워드" ~/.claude/skills/`
```

### 변형 C: 강조 스타일

```xml
<available_skills>
검증된 기법, 패턴, 도구의 개인 라이브러리가
`~/.claude/skills/` 에 있습니다.

카테고리 탐색: `ls ~/.claude/skills/`
검색: `grep -r "키워드" ~/.claude/skills/ --include="SKILL.md"`

사용법: `skills/using-my-poor-ai`
</available_skills>

<important_info_about_skills>
Claude는 작업에 어떻게 접근할지 안다고 생각할 수 있지만, 스킬
라이브러리에는 흔한 실수를 방지하는 검증된 접근법이 담겨 있습니다.

이것은 매우 중요합니다. 어떤 작업이든 시작하기 전에 스킬을 확인하세요!

프로세스:
1. 작업 시작? 확인: `ls ~/.claude/skills/[카테고리]/`
2. 스킬 발견? 진행하기 전에 완전히 읽기
3. 스킬의 안내를 따르기 — 알려진 함정을 방지합니다

당신의 작업에 스킬이 존재했는데 사용하지 않았다면, 실패한 것입니다.
</important_info_about_skills>
```

### 변형 D: 프로세스 중심

```markdown
## 스킬 활용

모든 작업에 대한 워크플로우:

1. **시작 전:** 관련 스킬 확인
   - 탐색: `ls ~/.claude/skills/`
   - 검색: `grep -r "증상" ~/.claude/skills/`

2. **스킬이 존재하면:** 진행하기 전에 완전히 읽기

3. **스킬을 따르기** — 과거 실패에서 얻은 교훈이 담겨 있습니다

스킬 라이브러리는 흔한 실수를 반복하지 않게 합니다.
시작 전에 확인하지 않는 것은 그 실수를 반복하기로 선택하는 것입니다.

여기서 시작: `skills/using-my-poor-ai`
```

## 테스트 프로토콜

각 변형에 대해:

1. **NULL 기준 먼저 실행** (스킬 문서 없음)
   - 에이전트가 어떤 선택지를 고르는지 기록
   - 정확한 합리화 포착

2. **변형 실행** (동일한 시나리오로)
   - 에이전트가 스킬을 확인하는가?
   - 발견하면 스킬을 사용하는가?
   - 위반하면 합리화 포착

3. **압박 테스트** — 시간/매몰 비용/권위 추가
   - 압박 하에서도 에이전트가 여전히 확인하는가?
   - 준수가 무너지는 지점을 문서화

4. **메타 테스트** — 에이전트에게 문서 개선 방법 질문
   - "문서가 있었는데 확인하지 않았다. 왜인가?"
   - "문서를 어떻게 더 명확하게 만들 수 있는가?"

## 성공 기준

**변형이 성공하는 경우:**

- 에이전트가 요청 없이 스킬을 확인
- 에이전트가 행동 전에 스킬을 완전히 읽음
- 에이전트가 압박 하에서도 스킬 안내를 따름
- 에이전트가 준수를 합리화로 회피하지 못함

**변형이 실패하는 경우:**

- 압박이 없어도 에이전트가 확인을 건너뜀
- 에이전트가 읽지 않고 "개념을 적용"
- 에이전트가 압박 하에서 합리화로 회피
- 에이전트가 스킬을 요구사항이 아닌 참조로 취급

## 예상 결과

**NULL:** 에이전트가 가장 빠른 경로를 선택, 스킬 인식 없음

**변형 A:** 압박이 없으면 확인할 수도, 압박 하에서는 건너뜀

**변형 B:** 가끔 확인, 합리화로 회피하기 쉬움

**변형 C:** 강한 준수, 다만 너무 경직돼 보일 수 있음

**변형 D:** 균형적이지만 더 김 — 에이전트가 내재화할 것인가?
