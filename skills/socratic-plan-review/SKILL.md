# Socratic Plan Review — 소크라테스 산파술 플랜 검증

## Overview

플랜은 이미 존재하지만 "미완성 태아" 상태임. 산파술(maieutics)은 플랜을 직접 수정하는 것이 아니라, **7개 카테고리의 구조화된 질문**으로 숨겨진 가정·미검토 엣지 케이스·누락된 롤백 시나리오를 표면화하여 플랜이 완전한 형태로 "탄생"하도록 도움.

> **산파술 원칙**: 답을 대신 써주는 것이 아님. 질문이 플랜을 완성시킴.

## 발동 조건

다음 중 하나라도 해당하면 이 스킬을 적용함:

- 아키텍처 변경 (모듈·레이어·인터페이스 구조 변경)
- 3개 이상 파일 수정
- 비가역 작업 (삭제, 마이그레이션, force push 등)
- 신규 외부 의존성 도입

서브에이전트 자기 검토만으로는 불충분하다고 판단될 때도 사용.

## 7개 질문 카테고리

플랜 검증 전 아래 7개 카테고리를 순서대로 점검함.
각 카테고리에서 "예, 문제없다"는 답이 나오려면 구체적 근거가 있어야 함.

### 1. 가정 검증 (Assumption Audit)

- 이 플랜의 핵심 전제는 무엇인가?
- 그 전제가 틀릴 가능성은?
- 전제가 무너지면 플랜의 어느 단계가 가장 먼저 깨지는가?

### 2. 엣지 케이스 (Edge Case Probe)

- 입력이 비어 있거나 null일 때?
- 동시 접근이 발생할 때?
- 데이터가 예상 범위를 벗어날 때?
- 네트워크/디스크/메모리가 실패할 때?

### 3. 롤백 시나리오 (Rollback Mapping)

- N번째 단계 실패 시 롤백 경로는?
- 중간 상태(half-applied)가 생길 수 있는 지점은?
- 롤백 후 시스템 상태가 일관성을 유지하는가?

### 4. 대안 탐색 (Alternative Examination)

- 다른 접근 방법은 무엇이 있는가?
- 현재 방법을 선택한 이유는?
- 탈락한 대안들의 근거가 문서화되어 있는가?

### 5. 의존성 분석 (Dependency Risk)

- 새로 도입되는 외부 의존성 목록은?
- 각 의존성이 실패하거나 변경될 때 영향은?
- 의존성 버전 고정·업그레이드 정책은?

### 6. 영향 범위 (Impact Scope)

- 이 변경이 직접 영향을 주는 컴포넌트는?
- 간접 영향(downstream)을 받는 곳은?
- 테스트 스위트에서 이 범위가 커버되는가?

### 7. 검증 가능성 (Verifiability)

- 완료를 어떻게 확인할 것인가? (테스트·로그·지표)
- 성공 기준이 구체적이고 측정 가능한가?
- 부분 성공과 완전 성공을 어떻게 구분하는가?

## 5 Whys for Plans

설계 결정에 확신이 없거나 가정 검증이 불충분하면 해당 결정에 "왜?"를 5번 반복함.

```
왜 이 방법을 선택했는가? → (답)
왜 그것이 맞는가?         → (답)
왜 대안이 열등한가?        → (답)
왜 이 타이밍인가?          → (답)
왜 이 범위인가?            → (답)
```

5번 "왜" 후에도 근거가 견고하면 진행. 모호해지면 플랜을 보강하거나 사용자에게 확인.

## 완료 기준

다음을 모두 충족해야 산파술 완료:

- [ ] 7개 카테고리 모두 검토됨
- [ ] 각 카테고리에 구체적 답변 (단순 "문제없음" 불가)
- [ ] 롤백 경로가 최소 1개 이상 명시됨
- [ ] 대안 접근이 1개 이상 검토되고 탈락 근거가 있음
- [ ] 성공 검증 방법이 구체적으로 기술됨

## Golden Rules

1. **질문이 완성시킴** — 답을 대신 써주지 않음. 플랜의 빈 자리를 질문으로 채움.
2. **"괜찮다"는 증거를 요구함** — 근거 없는 "문제없음"은 수용하지 않음.
3. **중간 상태를 두려워함** — 부분 적용(half-applied) 상태가 가장 위험함.
4. **대안은 반드시 검토함** — 첫 번째 떠오른 접근이 최선이 아닐 확률이 높음.
