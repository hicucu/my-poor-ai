# 플랜 작성

## 개요

코드베이스에 대한 컨텍스트가 전혀 없고 취향이 의심스러운 엔지니어를 가정하고 종합적인 구현 플랜을 작성. 각 작업에서 어떤 파일을 다뤄야 하는지, 코드, 테스트, 확인이 필요한 문서, 테스트 방법 등 필요한 모든 것을 문서화. 전체 플랜을 한 입에 먹기 좋은 크기의 작업으로 제공. DRY. YAGNI. TDD. 잦은 커밋.

숙련된 개발자이지만 우리의 툴셋이나 문제 도메인에 대해 거의 모른다고 가정. 좋은 테스트 설계에 대해 잘 모른다고 가정.

**시작 시 공지:** "플랜 작성 skill을 사용하여 구현 플랜을 작성함."

**컨텍스트:** 격리된 worktree에서 작업 중인 경우, 실행 시점에 `my-poor-ai:using-git-worktrees` skill을 통해 생성되어 있어야 함.

**플랜 저장 위치:** `_workspaces/{branch-slug}/specs/spec-{a,b,...}.md`

- (플랜 위치에 대한 사용자 설정이 있으면 이 기본값보다 우선)

## 범위 확인

스펙이 여러 독립적인 서브시스템을 다루는 경우, 브레인스토밍 단계에서 서브 프로젝트 스펙으로 분리되었어야 함. 그렇지 않다면, 별도의 플랜으로 분리할 것을 제안 — 서브시스템별로 하나씩. 각 플랜은 그 자체로 동작하고 테스트 가능한 소프트웨어를 산출해야 함.

## 파일 구조

작업을 정의하기 전에, 생성하거나 수정할 파일과 각 파일의 역할을 파악. 여기서 분해 결정이 확정됨.

- 명확한 경계와 잘 정의된 인터페이스를 가진 단위로 설계. 각 파일은 하나의 명확한 책임을 가져야 함.
- 한 번에 컨텍스트에 담을 수 있는 코드에 대해 가장 잘 추론할 수 있으며, 파일이 집중되어 있을 때 편집이 더 안정적. 너무 많은 일을 하는 큰 파일보다 더 작고 집중된 파일을 선호.
- 함께 변경되는 파일은 함께 위치. 기술 계층이 아닌 책임별로 분리.
- 기존 코드베이스에서는 확립된 패턴을 따를 것. 코드베이스에서 큰 파일을 사용한다면 일방적으로 구조를 바꾸지 말 것 — 단, 수정 중인 파일이 다루기 어려울 정도로 커졌다면 플랜에 분리를 포함하는 것이 합리적.

이 구조가 작업 분해에 영향을 줌. 각 작업은 독립적으로 의미 있는 자기 완결적 변경을 산출해야 함.

## 한 입 크기 작업 세분화

**각 단계는 하나의 행동 (2~5분):**

- "실패하는 테스트 작성" — 단계
- "실패하는지 실행하여 확인" — 단계
- "테스트를 통과시키는 최소한의 코드 구현" — 단계
- "테스트 실행하여 통과 확인" — 단계
- "커밋" — 단계

## 플랜 문서 헤더

**모든 플랜은 반드시 이 헤더로 시작해야 함:**

```markdown
# [기능 이름] 구현 플랜

> **에이전트 작업자를 위해:** 필수 서브 SKILL: my-poor-ai:subagent-driven-development (권장) 또는 my-poor-ai:executing-plans를 사용하여 이 플랜을 작업 단위로 구현. 단계는 추적을 위해 체크박스 (`- [ ]`) 문법 사용.

**목표:** [무엇을 만드는지 한 문장으로]

**아키텍처:** [접근 방식에 대해 2~3 문장]

**기술 스택:** [핵심 기술/라이브러리]

---
```

## 작업 구조

> 아래는 언어 중립 의사코드 예시. 실제 작성 시에는 프로젝트 언어/프레임워크의 문법으로 바꿔 쓰되, 단계의 입자도와 코드 블록 필수 원칙은 그대로 유지. 테스트·구현 실행 명령은 `_workspaces/stack-profile.json`의 `testFramework`가 결정하는 명령으로 채울 것 — 특정 스택을 임의로 고정하지 말 것.

````markdown
### 작업 N: [컴포넌트 이름]

**파일:**

- 생성: `<프로젝트 규칙에 따른 정확한 소스 파일 경로>`
- 수정: `<정확한 기존 파일 경로>:123-145`
- 테스트: `<프로젝트 규칙에 따른 정확한 테스트 파일 경로>`

- [ ] **단계 1: 실패하는 테스트 작성**

```
# 의사코드 — 실제 테스트 프레임워크 문법으로 작성
test "특정 동작":
    result = function(input)
    assert result == expected
```

- [ ] **단계 2: 테스트 실행하여 실패 확인**

실행: `<stack-profile.json의 testFramework가 결정하는 단일 테스트 실행 명령>`
예상 결과: "function 정의되지 않음"으로 FAIL

- [ ] **단계 3: 최소 구현 작성**

```
# 의사코드 — 실제 언어 문법으로 작성
function(input):
    return expected
```

- [ ] **단계 4: 테스트 실행하여 통과 확인**

실행: `<stack-profile.json의 testFramework가 결정하는 동일한 테스트 실행 명령>`
예상 결과: PASS

- [ ] **단계 5: 커밋**

```bash
git add <변경한 테스트 파일> <변경한 소스 파일>
git commit -m "feat: add specific feature"
```
````

## Phase 그룹 (선택)

작업이 많고 자연스러운 마일스톤이 있는 **큰 스펙에서만**, 작업들을 `## Phase N: [이름]` 헤더로 묶을 수 있음. 각 Phase 헤더 바로 아래에 완료 체크박스를 둠:

```markdown
## Phase 1: [마일스톤 이름]

- [ ] **Phase 1 완료** (이 그룹의 모든 작업 완료 시 체크)

  ### 작업 1: ...

  ### 작업 2: ...
```

작은 스펙은 Phase 없이 작업만 나열. **새 고정 계층을 강제하지 않음** — 마일스톤이 뚜렷할 때만 사용. 현재 구조(spec → 작업 → 단계)는 그대로 유지되며 Phase는 선택적 그룹일 뿐.

## 플레이스홀더 금지

모든 단계에는 엔지니어가 필요로 하는 실제 내용이 포함되어야 함. 다음은 **플랜 실패** — 절대 작성 금지:

- "TBD", "TODO", "나중에 구현", "세부사항 채울 것"
- "적절한 에러 처리 추가" / "유효성 검사 추가" / "엣지 케이스 처리"
- "위에 대한 테스트 작성" (실제 테스트 코드 없이)
- "작업 N과 유사" (코드 반복 필요 — 엔지니어가 작업을 순서 없이 읽을 수 있음)
- 방법을 보여주지 않고 무엇을 할지만 설명하는 단계 (코드 단계에는 코드 블록 필수)
- 어느 작업에서도 정의되지 않은 타입, 함수, 메서드 참조

## 기억할 것

- 항상 정확한 파일 경로
- 모든 단계에 완전한 코드 — 단계가 코드를 변경하면 코드를 보여줄 것
- 예상 출력이 포함된 정확한 명령어
- DRY, YAGNI, TDD, 잦은 커밋

## 자체 검토

완전한 플랜 작성 후, 새로운 시각으로 스펙을 보고 플랜을 점검. 이것은 스스로 실행하는 체크리스트 — 서브에이전트 배포가 아님.

**1. 스펙 커버리지:** 스펙의 각 섹션/요구사항을 훑어볼 것. 그것을 구현하는 작업을 찾을 수 있는가? 누락된 부분 목록화.

**2. 플레이스홀더 스캔:** 플랜에서 위험 신호 검색 — "플레이스홀더 금지" 섹션의 패턴 중 하나라도. 수정할 것.

**3. 타입 일관성:** 이후 작업에서 사용한 타입, 메서드 시그니처, 프로퍼티 이름이 이전 작업에서 정의한 것과 일치하는가? 작업 3에서 `clearLayers()`로 호출되고 작업 7에서 `clearFullLayers()`로 호출되면 버그.

**4. 엔지니어가 이해할 수 있어야 함!** 엔지니어가 이해할 수 없는 추상화, 설계 패턴, 아키텍처를 도입하지 말 것.

**5. 단순한 구현 우선:** 스펙 요구사항을 충족하는 가장 단순한 구현을 선택. 엔지니어가 이해할 수 있는 수준의 단순함을 유지.

**6. 반복되는 코드:** 반복되는 코드가 있다면, 그것을 재사용 가능한 함수/메서드로 추출할 것. 그러나 과도한 추상화는 피할 것.

**7. 간결한 코드:** 적은 코드로 동일한 기능을 구현할 수 있는 방법이 있는지 검토. 불필요한 복잡성을 제거. 그러나 가독성을 해치지 않도록 주의.

**8. 표준 라이브러리 활용:** 표준 라이브러리나 프레임워크의 기능으로 해결할 수 있다면 표준 라이브러리를 활용하여 구현을 단순화. 그러나 표준 라이브러리의 사용이 과도한 코드를 생성하거나 가독성을 해친다면 피할 것.

**9. 나중을 위한 보일러플레이트 금지** 계획에 없는 나중을 위한 보일러플레이트 코드를 작성하지 말 것.

**10. 절대적으로 지켜야 할 것** 신뢰 경계를 벗어나지 않도록 주의. 데이터 손실 방어/보안 조치/기본 접근성/명시적 요청된 기능 등, 스펙에서 요구하는 최소한의 신뢰 경계는 반드시 지킬 것.

문제를 발견하면 인라인으로 수정. 재검토할 필요 없음 — 수정하고 계속. 작업이 없는 스펙 요구사항을 발견하면 작업을 추가할 것.

## 실행 인계

플랜 저장 후, 실행 방식 선택을 제안:

**"플랜이 완성되어 `_workspaces/{branch-slug}/specs/<filename>.md`에 저장됨. 두 가지 실행 옵션 있음:**

**1. 서브에이전트 주도 방식 (권장)** — 작업별로 새로운 서브에이전트를 배포하고, 작업 사이에 검토하며 빠른 반복

**2. 인라인 실행** — executing-plans를 사용하여 이 세션에서 작업 실행, 체크포인트가 있는 배치 실행

**어떤 방식을 선택하시겠습니까?"**

**서브에이전트 주도 방식 선택 시:**

- **필수 서브 SKILL:** my-poor-ai:subagent-driven-development 사용
- 작업별 새 서브에이전트 + 2단계 검토

**인라인 실행 선택 시:**

- **필수 서브 SKILL:** my-poor-ai:executing-plans 사용
- 검토 체크포인트가 있는 배치 실행

## HANDOFF.md — 세션 인계 안내 (실행 주체에게)

복잡 경로에서 컨텍스트가 끊겨도 다음 세션·에이전트가 이어받을 수 있도록, 실행 주체(subagent-driven-development / executing-plans)는 `_workspaces/{branch-slug}/HANDOFF.md`를 유지해야 함. writing-plans는 이 파일을 직접 만들지 않으며, 아래 규약을 실행 주체에 전달함:

- **갱신 트리거**: spec 완료 시 또는 phase 완료 시. **task 완료는 체크박스로만 추적 — HANDOFF를 건드리지 않음.**
- **없으면 생성**: HANDOFF.md가 없는 상태로 갱신 트리거가 오면 먼저 생성 후 채움.
- **형식**: 본문은 최신 상태로 덮어쓰기, 하단 인계 로그는 **최근 5개만** 유지.
- **범위**: 복잡 경로(FULL) 전용. 단순·디버깅 경로에는 만들지 않음.

본문 템플릿:

```markdown
# HANDOFF: {branch-slug}

**갱신:** {YYYY-MM-DD} · **현재 위치:** {spec-x / Phase n}

## 지금까지 (무엇을·왜)

## 현재 진행 중

## 다음 이어받을 일

## 주의·막힌 점·가정

## 참조

- design.md · specs/ · pipeline-state.md

---

## 인계 로그 (최근 5개, 최신 위)

- {YYYY-MM-DD} {spec-x} 완료
```
