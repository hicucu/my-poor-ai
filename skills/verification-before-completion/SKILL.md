# 완료 전 검증

## 개요

검증하지 않고 작업이 끝났다고 주장하는 것은 효율적인 게 아니라 부정직한 것임.

**핵심 원칙:** 항상 주장보다 증거가 먼저다.

**규칙의 문자를 위반하는 것은 규칙의 정신을 위반하는 것임.**

## 철칙

```
최신 검증 증거 없이 완료 주장 금지
```

지금 이 메시지에서 검증 명령을 직접 실행하지 않았다면, 통과한다고 주장할 수 없음.

## 게이트 함수

```
어떤 상태를 주장하거나 만족감을 표현하기 전에:

1. 식별: 이 주장을 증명하는 명령어는 무엇인가?
2. 실행: 전체 명령어를 실행하라 (새로, 완전하게)
3. 읽기: 전체 출력을 읽고, 종료 코드를 확인하고, 실패 횟수를 세어라
4. 검증: 출력이 주장을 확인하는가?
   - 아니오인 경우: 증거와 함께 실제 상태를 기술하라
   - 예인 경우: 증거와 함께 주장하라
5. 그런 다음에만: 주장하라

단계를 건너뛰는 것 = 검증이 아닌 거짓말
```

**GOAL.md가 있으면:** 위 게이트 함수를 GOAL.md의 각 성공 기준 항목에도 동일하게 적용함. 항목마다 증거로 대조하여 전부 충족됐을 때만 "목표 달성"을 주장함. (GOAL.md가 없으면 이 단계는 건너뜀.)

## 흔한 실패

| 주장             | 요건                        | 불충분한 것                 |
| ---------------- | --------------------------- | --------------------------- |
| 테스트 통과      | 테스트 명령 출력: 0 실패    | 이전 실행, "통과할 것 같다" |
| 린터 클린        | 린터 출력: 0 오류           | 부분 확인, 추론             |
| 빌드 성공        | 빌드 명령: exit 0           | 린터 통과, 로그가 좋아 보임 |
| 버그 수정        | 원래 증상 테스트: 통과      | 코드 변경, 수정 가정        |
| 회귀 테스트 작동 | RED-GREEN 사이클 검증       | 테스트 한 번 통과           |
| 에이전트 완료    | VCS diff가 변경 사항 표시   | 에이전트가 "성공" 보고      |
| 요구사항 충족    | 항목별 체크리스트           | 테스트 통과                 |
| 목표 달성        | GOAL.md 성공 기준 전부 충족 | 일부만 충족, 추정           |

## 위험 신호 - 멈춤

- "~일 것이다", "아마", "~인 것 같다" 같은 표현 사용
- 검증 전에 만족감 표현 ("좋아!", "완벽해!", "완료!", 등)
- 검증 없이 커밋/푸시/PR 직전
- 에이전트 성공 보고를 신뢰하는 것
- 부분 검증에 의존하는 것
- "이번 한 번만" 생각
- 피곤하여 작업을 끝내고 싶은 것
- **검증 실행 없이 성공을 암시하는 모든 표현**

## 합리화 방지

| 변명                                     | 현실                            |
| ---------------------------------------- | ------------------------------- |
| "이제 작동할 것이다"                     | 검증 실행                       |
| "확신한다"                               | 확신 ≠ 증거                     |
| "이번 한 번만"                           | 예외 없음                       |
| "린터가 통과했다"                        | 린터 ≠ 컴파일러                 |
| "에이전트가 성공이라고 했다"             | 독립적으로 검증                 |
| "피곤하다"                               | 피로 ≠ 변명                     |
| "부분 확인으로 충분하다"                 | 부분은 아무것도 증명하지 않음   |
| "다른 표현이라서 규칙이 적용되지 않는다" | 문자보다 정신이 우선            |

## 주요 패턴

**테스트:**

```
✅ [테스트 명령 실행] [확인: 34/34 통과] "모든 테스트 통과"
❌ "이제 통과할 것이다" / "올바르게 보인다"
```

**회귀 테스트 (TDD RED-GREEN):**

```
✅ 작성 → 실행(통과) → 수정 되돌리기 → 실행(반드시 실패) → 복원 → 실행(통과)
❌ "회귀 테스트를 작성했다" (RED-GREEN 검증 없이)
```

**빌드:**

```
✅ [빌드 실행] [확인: exit 0] "빌드 통과"
❌ "린터가 통과했다" (린터는 컴파일을 확인하지 않음)
```

**요구사항:**

```
✅ 계획 재검토 → 체크리스트 작성 → 각 항목 검증 → 누락 또는 완료 보고
❌ "테스트 통과, 단계 완료"
```

**목표 (GOAL.md):**

```
✅ GOAL.md 성공 기준 항목별로 실제 증거 대조 → 전부 충족 확인 후 완료 주장
❌ "목표 달성" (성공 기준 일부만 확인, 추정)
```

**에이전트 위임:**

```
✅ 에이전트가 성공 보고 → VCS diff 확인 → 변경사항 검증 → 실제 상태 보고
❌ 에이전트 보고를 신뢰하는 것
```

## 중요한 이유

24건의 실패 경험으로부터:

- 파트너가 "믿지 않겠다"고 말함 - 신뢰 붕괴
- 정의되지 않은 함수가 배포됨 - 크래시 발생
- 누락된 요구사항이 배포됨 - 불완전한 기능
- 거짓 완료로 시간 낭비 → 재지시 → 재작업
- 위반 사항: "정직은 핵심 가치임. 거짓말하면 교체될 것임."

## 적용 시점

**항상 다음 전에:**

- 성공/완료 주장의 모든 변형
- 모든 만족감 표현
- 작업 상태에 대한 모든 긍정적 진술
- 커밋, PR 생성, 작업 완료
- 다음 작업으로 이동
- 에이전트에게 위임

**규칙 적용 대상:**

- 정확한 문구
- 바꿔 말하기 및 동의어
- 성공의 암시
- 완료/정확성을 시사하는 모든 커뮤니케이션

## 결론

**검증에 지름길은 없음.**

명령을 실행할 것. 출력을 읽을 것. 그런 다음 결과를 주장할 것.

이것은 협상 불가능함.
