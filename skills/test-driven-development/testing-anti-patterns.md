# 테스트 안티패턴

**이 참조를 불러올 때:** 테스트를 작성·변경하거나, 모킹을 추가하거나, 프로덕션 코드에 테스트 전용 메서드를 넣고 싶을 때.

> 아래 예시는 **언어 중립 의사코드**임. 실제 문법(테스트 함수·단언·모킹 API)은 `stack-profile.json`의 `testFramework`가 결정함.

## 개요

테스트는 모킹의 동작이 아니라 **실제 동작**을 검증해야 함. 모킹은 격리하기 위한 수단이지, 검증 대상 자체가 아님.

**핵심 원칙:** 모킹이 하는 일이 아니라, 코드가 하는 일을 테스트할 것.

**엄격한 TDD는 아래 안티패턴들을 예방함.**

## 철칙

```
1. 모킹의 동작을 테스트하지 말 것
2. 프로덕션 클래스에 테스트 전용 메서드를 넣지 말 것
3. 의존성을 이해하지 않고 모킹하지 말 것
```

## 안티패턴 1: 모킹의 동작을 테스트

**위반:**

```
# ❌ 나쁨: 모킹이 존재한다는 사실을 테스트
test "사이드바를 렌더링한다":
    render(Page)
    assert 화면에 존재함(testId = "sidebar-mock")
```

**왜 잘못인가:**

- 컴포넌트가 동작하는지가 아니라 모킹이 동작하는지를 검증하고 있음
- 모킹이 있으면 통과하고 없으면 실패 — 실제 동작에 대해 아무것도 말해주지 않음

**검토 질문:** "지금 모킹의 동작을 테스트하고 있는가?"

**수정:**

```
# ✅ 좋음: 실제 컴포넌트를 테스트하거나, 모킹하지 않는다
test "사이드바를 렌더링한다":
    render(Page)                       # 사이드바를 모킹하지 않음
    assert 화면에 존재함(role = "navigation")

# 격리를 위해 사이드바를 반드시 모킹해야 한다면:
# 모킹 자체를 단언하지 말고, 사이드바가 있는 상태에서 Page의 동작을 테스트한다
```

### 게이트 함수

```
모킹 요소에 단언하기 전에:
  질문: "실제 컴포넌트 동작을 테스트하는가, 모킹의 존재를 테스트하는가?"

  모킹의 존재를 테스트한다면:
    멈춤 — 단언을 삭제하거나 컴포넌트의 모킹을 해제하라

  실제 동작을 대신 테스트하라
```

## 안티패턴 2: 프로덕션에 테스트 전용 메서드

**위반:**

```
# ❌ 나쁨: destroy()가 테스트에서만 쓰임
class Session:
    method destroy():            # 프로덕션 API처럼 보인다!
        workspaceManager.destroyWorkspace(this.id)
        # ... 정리 로직

# 테스트에서
afterEach: session.destroy()
```

**왜 잘못인가:**

- 프로덕션 클래스가 테스트 전용 코드로 오염됨
- 프로덕션에서 실수로 호출되면 위험함
- YAGNI와 관심사 분리를 위반함
- 객체 수명과 엔티티 수명을 혼동함

**수정:**

```
# ✅ 좋음: 테스트 정리는 테스트 유틸리티가 담당
# Session에는 destroy()가 없다 — 프로덕션에서는 상태가 없다

# test-utils 에서
function cleanupSession(session):
    workspace = session.getWorkspaceInfo()
    if workspace:
        workspaceManager.destroyWorkspace(workspace.id)

# 테스트에서
afterEach: cleanupSession(session)
```

### 게이트 함수

```
프로덕션 클래스에 메서드를 추가하기 전에:
  질문: "이건 테스트에서만 쓰이는가?"
  예라면:
    멈춤 — 추가하지 말고 테스트 유틸리티에 넣어라

  질문: "이 클래스가 이 리소스의 수명을 소유하는가?"
  아니오라면:
    멈춤 — 이 메서드가 들어갈 클래스가 아니다
```

## 안티패턴 3: 이해 없이 모킹

**위반:**

```
# ❌ 나쁨: 모킹이 테스트 로직을 망가뜨림
test "중복 서버를 감지한다":
    # 테스트가 의존하는 설정 기록을 모킹이 막아버린다!
    mock(ToolCatalog.discoverAndCacheTools) returns 없음

    addServer(config)
    addServer(config)   # 예외를 던져야 하는데 — 던지지 않는다!
```

**왜 잘못인가:**

- 모킹한 메서드에 테스트가 의존하던 부수효과(설정 기록)가 있었음
- "안전하게" 하려는 과도한 모킹이 실제 동작을 망가뜨림
- 테스트가 잘못된 이유로 통과하거나, 알 수 없게 실패함

**수정:**

```
# ✅ 좋음: 올바른 수준에서 모킹
test "중복 서버를 감지한다":
    # 느린 부분만 모킹하고, 테스트가 필요로 하는 동작은 보존한다
    mock(MCPServerManager)   # 느린 서버 기동만 모킹

    addServer(config)        # 설정 기록됨
    addServer(config)        # 중복 감지됨 ✓
```

### 게이트 함수

```
어떤 메서드든 모킹하기 전에:
  멈춤 — 아직 모킹하지 말 것

  1. 질문: "실제 메서드는 어떤 부수효과를 가지는가?"
  2. 질문: "이 테스트가 그 부수효과 중 하나에 의존하는가?"
  3. 질문: "이 테스트가 무엇을 필요로 하는지 완전히 이해했는가?"

  부수효과에 의존한다면:
    더 낮은 수준(실제로 느린/외부 연산)에서 모킹하라
    또는 필요한 동작을 보존하는 테스트 더블을 쓰라
    테스트가 의존하는 상위 메서드를 모킹하지 말 것

  테스트가 무엇에 의존하는지 모르겠다면:
    실제 구현으로 테스트를 먼저 실행하라
    무엇이 실제로 일어나야 하는지 관찰하라
    그 다음 올바른 수준에서 최소한의 모킹을 추가하라

  위험 신호:
    - "안전하게 이것도 모킹하자"
    - "느릴 수 있으니 모킹하는 게 낫겠다"
    - 의존성 사슬을 이해하지 못한 채 모킹
```

## 안티패턴 4: 불완전한 모킹

**위반:**

```
# ❌ 나쁨: 부분 모킹 — 필요하다고 생각한 필드만
mockResponse = {
    status: "success",
    data: { userId: "123", name: "Alice" }
    # 누락: 하위 코드가 사용하는 metadata
}

# 나중에: 코드가 response.metadata.requestId 에 접근할 때 깨진다
```

**왜 잘못인가:**

- **부분 모킹은 구조적 가정을 숨긴다** — 아는 필드만 모킹했을 뿐임
- **하위 코드가 포함하지 않은 필드에 의존할 수 있음** — 조용한 실패
- **테스트는 통과하지만 통합에서 실패함** — 모킹은 불완전, 실제 API는 완전
- **거짓 확신** — 테스트가 실제 동작에 대해 아무것도 증명하지 못함

**철의 규칙:** 당장의 테스트가 쓰는 필드만이 아니라, **실제 존재하는 그대로의 완전한 데이터 구조**를 모킹할 것.

**수정:**

```
# ✅ 좋음: 실제 API의 완전성을 그대로 반영
mockResponse = {
    status: "success",
    data: { userId: "123", name: "Alice" },
    metadata: { requestId: "req-789", timestamp: 1234567890 }
    # 실제 API가 반환하는 모든 필드
}
```

### 게이트 함수

```
모킹 응답을 만들기 전에:
  확인: "실제 API 응답에는 어떤 필드가 들어 있는가?"

  행동:
    1. 문서/예시에서 실제 API 응답을 살펴보라
    2. 하위에서 소비될 수 있는 모든 필드를 포함하라
    3. 모킹이 실제 응답 스키마와 완전히 일치하는지 확인하라

  핵심:
    모킹을 만든다면, 전체 구조를 이해해야 한다
    부분 모킹은 코드가 생략된 필드에 의존할 때 조용히 실패한다

  불확실하면: 문서에 있는 모든 필드를 포함하라
```

## 안티패턴 5: 통합 테스트를 나중에

**위반:**

```
✅ 구현 완료
❌ 테스트 미작성
"테스트할 준비됨"
```

**왜 잘못인가:**

- 테스트는 구현의 일부이지, 선택적 후속 작업이 아님
- TDD였다면 이를 잡아냈을 것임
- 테스트 없이 완료를 주장할 수 없음

**수정:**

```
TDD 사이클:
1. 실패하는 테스트 작성
2. 통과하도록 구현
3. 리팩터링
4. 그 다음에 완료 주장
```

## 모킹이 너무 복잡해질 때

**경고 신호:**

- 모킹 설정이 테스트 로직보다 긺
- 테스트를 통과시키려고 모든 것을 모킹함
- 실제 컴포넌트에 있는 메서드가 모킹에는 없음
- 모킹이 바뀌면 테스트가 깨짐

**검토 질문:** "여기서 모킹이 정말 필요한가?"

**고려할 것:** 실제 컴포넌트로 하는 통합 테스트가 복잡한 모킹보다 더 단순한 경우가 많음.

## TDD가 이 안티패턴들을 예방함

**TDD가 돕는 이유:**

1. **테스트 먼저 작성** → 실제로 무엇을 테스트하는지 생각하게 강제함
2. **실패를 직접 확인** → 모킹이 아니라 실제 동작을 테스트함을 확인함
3. **최소 구현** → 테스트 전용 메서드가 끼어들지 않음
4. **실제 의존성** → 모킹하기 전에 테스트가 실제로 무엇을 필요로 하는지 보게 됨

**모킹의 동작을 테스트하고 있다면, TDD를 위반한 것임** — 실제 코드에 대해 테스트가 실패하는 것을 먼저 보지 않고 모킹을 추가한 것임.

## 빠른 참조

| 안티패턴                      | 수정                                   |
| ----------------------------- | -------------------------------------- |
| 모킹 요소에 단언              | 실제 컴포넌트를 테스트하거나 모킹 해제 |
| 프로덕션의 테스트 전용 메서드 | 테스트 유틸리티로 이동                 |
| 이해 없는 모킹                | 의존성 먼저 이해, 최소한으로 모킹      |
| 불완전한 모킹                 | 실제 API를 완전히 반영                 |
| 테스트를 나중에               | TDD — 테스트 먼저                      |
| 과도하게 복잡한 모킹          | 통합 테스트를 고려                     |

## 위험 신호

- 단언이 `*-mock` 류 testId를 확인함
- 테스트 파일에서만 호출되는 메서드
- 모킹 설정이 테스트의 절반을 넘음
- 모킹을 제거하면 테스트가 깨짐
- 모킹이 왜 필요한지 설명하지 못함
- "안전하게" 모킹함

## 결론

**모킹은 격리하는 도구이지, 테스트하는 대상이 아님.**

TDD가 모킹의 동작을 테스트하고 있음을 드러낸다면, 무언가 잘못된 것임.

수정: 실제 동작을 테스트하거나, 애초에 왜 모킹하는지를 되물을 것.
