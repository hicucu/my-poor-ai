# 체계적 디버깅

## 개요

무작위 수정은 시간을 낭비하고 새로운 버그를 만듦. 빠른 패치는 근본적인 문제를 감춤.

**핵심 원칙:** 수정 시도 전에 항상 근본 원인을 찾을 것. 증상 수정은 실패.

**이 프로세스의 형식을 위반하는 것은 디버깅의 정신을 위반하는 것.**

## 철칙

```
근본 원인 조사 없이 수정 금지
```

1단계를 완료하지 않았다면 수정을 제안할 수 없음.

## 사용 시점

모든 기술적 이슈에 사용:

- 테스트 실패
- 프로덕션 버그
- 예상치 못한 동작
- 성능 문제
- 빌드 실패
- 통합 이슈

**특히 다음 경우에 사용:**

- 시간 압박이 있을 때 (긴급 상황은 추측을 유발)
- "빠른 수정 하나"가 명확해 보일 때
- 이미 여러 수정을 시도했을 때
- 이전 수정이 동작하지 않았을 때
- 이슈를 완전히 이해하지 못했을 때

**다음 경우에도 생략 금지:**

- 이슈가 단순해 보일 때 (단순한 버그도 근본 원인이 있음)
- 급할 때 (서두르면 재작업이 보장됨)
- 즉시 수정을 원할 때 (체계적 접근이 무작정 고치는 것보다 빠름)

## 4단계 프로세스

다음 단계로 진행하기 전에 각 단계를 반드시 완료해야 함.

### 1단계: 근본 원인 조사

**수정 시도 전에:**

1. **에러 메시지를 주의 깊게 읽기**
   - 에러나 경고를 무시하지 말 것
   - 종종 정확한 해결책을 담고 있음
   - 스택 트레이스를 완전히 읽을 것
   - 줄 번호, 파일 경로, 에러 코드 메모

2. **일관되게 재현하기**
   - 신뢰할 수 있게 발생시킬 수 있는가?
   - 정확한 단계는 무엇인가?
   - 매번 발생하는가?
   - 재현 불가능한 경우 → 더 많은 데이터 수집, 추측 금지

3. **최근 변경사항 확인**
   - 무엇이 바뀌어 이것을 유발할 수 있는가?
   - Git diff, 최근 커밋
   - 새 의존성, 설정 변경
   - 환경 차이

4. **다중 컴포넌트 시스템에서 증거 수집**

   **시스템에 여러 컴포넌트가 있는 경우 (CI → 빌드 → 서명, API → 서비스 → 데이터베이스):**

   **수정 제안 전에 진단 계측 추가:**

   ```
   각 컴포넌트 경계마다:
     - 컴포넌트에 입력되는 데이터 로깅
     - 컴포넌트에서 출력되는 데이터 로깅
     - 환경/설정 전파 검증
     - 각 레이어의 상태 확인

   어디서 끊어지는지 보여주는 증거를 수집하기 위해 한 번 실행
   그런 다음 증거를 분석하여 실패한 컴포넌트 식별
   그런 다음 해당 특정 컴포넌트 조사
   ```

   **예시 (다중 레이어 시스템, 언어 중립):**

   ```
   # 각 컴포넌트 경계에 계측을 넣고 한 번 실행한다
   레이어 1 (진입점):     필요한 입력·시크릿이 존재하는지 로깅
   레이어 2 (처리/빌드):  환경·설정이 다음 단계로 전파됐는지 로깅
   레이어 3 (외부 연동):  연동 직전의 상태(자격증명·연결·세션) 로깅
   레이어 4 (실제 작업):  작업 호출과 그 결과를 상세 로깅
   ```

   **이를 통해 알 수 있음:** 어느 레이어가 실패하는지 (진입점 ✓, 처리 ✗)

5. **데이터 흐름 추적**

   **에러가 콜 스택 깊은 곳에 있는 경우:**

   완전한 역방향 추적 기법은 이 디렉토리의 `root-cause-tracing.md` 참조.

   **간략 버전:**
   - 잘못된 값은 어디서 발생하는가?
   - 무엇이 잘못된 값으로 이것을 호출했는가?
   - 출처를 찾을 때까지 계속 추적
   - 증상이 아닌 출처에서 수정

### 2단계: 패턴 분석

**수정 전에 패턴 찾기:**

1. **동작하는 예시 찾기**
   - 동일 코드베이스에서 유사하게 동작하는 코드 찾기
   - 깨진 것과 유사하게 동작하는 것은 무엇인가?

2. **참조와 비교**
   - 패턴을 구현 중이라면 참조 구현을 완전히 읽을 것
   - 훑어보지 말 것 — 모든 줄을 읽을 것
   - 패턴을 완전히 이해한 후 적용

3. **차이점 식별**
   - 동작하는 것과 깨진 것 사이의 차이는 무엇인가?
   - 아무리 작아도 모든 차이점 목록화
   - "그것은 중요하지 않을 것"이라고 가정하지 말 것

4. **의존성 이해**
   - 이것이 필요로 하는 다른 컴포넌트는 무엇인가?
   - 어떤 설정, 구성, 환경이 필요한가?
   - 어떤 가정을 하고 있는가?

### 3단계: 가설 및 테스트

**과학적 방법:**

1. **단일 가설 형성**
   - 명확하게 표현: "Y 때문에 X가 근본 원인이라고 생각함"
   - 적어둘 것
   - 모호하지 않고 구체적으로

2. **최소한으로 테스트**
   - 가설을 테스트하기 위해 가능한 가장 작은 변경
   - 한 번에 하나의 변수
   - 여러 가지를 동시에 수정하지 말 것

3. **계속하기 전에 검증**
   - 동작했는가? 예 → 4단계
   - 동작하지 않았는가? 새 가설 형성
   - 더 많은 수정을 추가하지 말 것

4. **모를 때**
   - "X를 이해하지 못함"이라고 말할 것
   - 아는 척하지 말 것
   - 도움 요청
   - 더 조사

### 4단계: 구현

**증상이 아닌 근본 원인 수정:**

1. **실패하는 테스트 케이스 생성**
   - 가능한 가장 단순한 재현
   - 가능하면 자동화된 테스트
   - 프레임워크가 없으면 일회성 테스트 스크립트
   - 수정 전에 반드시 있어야 함
   - 제대로 된 실패 테스트 작성을 위해 `my-poor-ai:test-driven-development` skill 사용

2. **단일 수정 구현**
   - 식별된 근본 원인 해결
   - 한 번에 하나의 변경
   - "하는 김에" 개선 없음
   - 묶음 리팩터링 없음

3. **수정 검증**
   - 이제 테스트가 통과하는가?
   - 다른 테스트가 깨지지 않았는가?
   - 이슈가 실제로 해결되었는가?

4. **수정이 동작하지 않는 경우**
   - 멈출 것
   - 세어볼 것: 몇 번의 수정을 시도했는가?
   - 3회 미만이면: 1단계로 돌아가서 새로운 정보로 재분석
   - **3회 이상이면: 멈추고 아키텍처에 의문을 제기 (아래 5단계)**
   - 아키텍처 논의 없이 4번째 수정 시도 금지

5. **3회 이상 수정 실패: 아키텍처에 의문 제기**

   **아키텍처 문제를 나타내는 패턴:**
   - 각 수정이 다른 곳에서 새로운 공유 상태/결합/문제를 드러냄
   - 수정에 "대규모 리팩터링"이 필요함
   - 각 수정이 다른 곳에서 새로운 증상을 만듦

   **멈추고 근본 원칙에 의문을 제기:**
   - 이 패턴이 근본적으로 건전한가?
   - "순전한 관성으로 유지"하고 있는가?
   - 계속 증상을 수정하는 것보다 아키텍처를 리팩터링해야 하는가?

   **더 많은 수정을 시도하기 전에 사용자와 논의**

   이것은 실패한 가설이 아님 — 잘못된 아키텍처.

## 위험 신호 — 멈추고 프로세스 따르기

다음을 생각하고 있다면:

- "지금은 빠른 수정, 나중에 조사"
- "X를 바꿔보고 동작하는지 보자"
- "여러 변경 추가하고 테스트 실행"
- "테스트 건너뛰고 수동으로 검증할게"
- "아마도 X일 것 같아서 수정해보자"
- "완전히 이해하지 못하지만 이게 동작할지도"
- "패턴은 X라고 하지만 다르게 적용하겠음"
- "주요 문제들은: [조사 없이 수정 목록]"
- 데이터 흐름 추적 전에 해결책 제안
- **"한 번 더 수정 시도" (이미 2회 이상 시도한 경우)**
- **각 수정이 다른 곳에서 새로운 문제를 드러냄**

**이 모든 것은 의미함: 멈출 것. 1단계로 돌아갈 것.**

**3회 이상 수정 실패:** 아키텍처에 의문 제기 (4단계 5번 참조)

## 사용자의 올바르지 않은 방향 신호

**다음 방향 전환에 주의:**

- "그게 일어나지 않는 건가요?" — 검증 없이 가정함
- "그게 보여줄 건가요...?" — 증거 수집을 추가했어야 함
- "추측하지 마세요" — 이해 없이 수정을 제안함
- "Ultrathink해보세요" — 증상이 아닌 근본 원칙에 의문 제기
- "막혔나요?" (좌절) — 접근 방식이 동작하지 않음

**이런 것들을 보면:** 멈출 것. 1단계로 돌아갈 것.

## 일반적인 합리화

| 변명                                    | 현실                                                             |
| --------------------------------------- | ---------------------------------------------------------------- |
| "이슈가 단순해서 프로세스가 필요 없음"  | 단순한 이슈도 근본 원인이 있음. 단순한 버그에는 프로세스가 빠름. |
| "긴급 상황이라 프로세스 시간 없음"      | 체계적 디버깅이 추측-확인 반복보다 빠름.                         |
| "먼저 이걸 시도해보고 조사하자"         | 첫 번째 수정이 패턴을 설정함. 처음부터 제대로 할 것.             |
| "수정이 동작하는지 확인 후 테스트 작성" | 테스트되지 않은 수정은 유지되지 않음. 테스트를 먼저 하면 증명됨. |
| "여러 수정을 동시에 하면 시간 절약"     | 무엇이 동작했는지 격리 불가. 새 버그 유발.                       |
| "참조가 너무 길어서 패턴을 적용하겠음"  | 부분적 이해는 버그를 보장함. 완전히 읽을 것.                     |
| "문제가 보여서 수정하겠음"              | 증상 파악 ≠ 근본 원인 이해.                                      |
| "한 번 더 수정 시도" (2회 이상 실패 후) | 3회 이상 실패 = 아키텍처 문제. 패턴에 의문 제기, 다시 수정 금지. |

## 빠른 참조

| 단계             | 핵심 활동                             | 성공 기준              |
| ---------------- | ------------------------------------- | ---------------------- |
| **1. 근본 원인** | 에러 읽기, 재현, 변경 확인, 증거 수집 | 무엇과 왜를 이해       |
| **2. 패턴**      | 동작하는 예시 찾기, 비교              | 차이점 식별            |
| **3. 가설**      | 이론 형성, 최소한으로 테스트          | 확인 또는 새 가설      |
| **4. 구현**      | 테스트 생성, 수정, 검증               | 버그 해결, 테스트 통과 |

## 프로세스가 "근본 원인 없음"을 드러낼 때

체계적 조사가 이슈가 진정으로 환경적, 타이밍 의존적, 또는 외부적임을 드러내면:

1. 프로세스를 완료한 것
2. 조사한 내용 문서화
3. 적절한 처리 구현 (재시도, 타임아웃, 에러 메시지)
4. 향후 조사를 위한 모니터링/로깅 추가

**단:** "근본 원인 없음" 케이스의 95%는 불완전한 조사.

## 지원 기법

이 기법들은 체계적 디버깅의 일부이며 이 디렉토리에서 사용 가능:

- **`root-cause-tracing.md`** — 원래 트리거를 찾기 위해 콜 스택을 역방향으로 버그 추적
- **`defense-in-depth.md`** — 근본 원인 발견 후 여러 레이어에 유효성 검사 추가
- **`condition-based-waiting.md`** — 임의 타임아웃을 조건 폴링으로 교체

**관련 skill:**

- **my-poor-ai:test-driven-development** — 실패하는 테스트 케이스 생성 (4단계 1번)
- **my-poor-ai:verification-before-completion** — 완료 주장 전 수정 동작 검증

## 실제 영향

디버깅 세션에서:

- 체계적 접근: 수정에 15~30분
- 무작위 수정 접근: 2~3시간의 헛수고
- 첫 번째 수정 성공률: 95% vs 40%
- 새로 도입된 버그: 거의 없음 vs 흔함
