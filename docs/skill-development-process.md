# 스킬 개발 프로세스

신규 스킬 작성부터 테스트 자산화까지의 표준 절차. 2026-07 `which-way-should-i-go` 개발·`socratic-plan-review` 점검 사례를 기반으로 정리함. `skills/writing-skills/SKILL.md`의 TDD 방법론을 실제 개발 흐름으로 구체화한 문서임.

## 전체 흐름

```
스킬 로드 → 맥락 탐색 → 의사결정 → 설계 승인 → TDD 사이클 → (내용 추가 반복)
→ 문서 정합성 확인 → 의존 스킬 역점검 → 테스트 자산화
```

## 1. 착수 규율 — 스킬 로드와 맥락 탐색

- `writing-skills` 로드 — "skills는 코드"라는 저장소 지침에 따라 작성 가이드라인부터 확보
- `brainstorming` 로드 — 창작 작업 전 필수 게이트 준수
- 기존 자산 중복 확인 — skills/agents/commands 전체에서 유사 기능 검색, 기존 스킬들의 frontmatter 컨벤션(영어 description, "Use when..." 형식, 한국어 본문) 조사

## 2. 의사결정과 설계 게이트

- 갈림길만 골라 사용자에게 질문 — 통합 방식(독립 스킬 vs 기존 스킬 수정), 이름, 산출물 형식 같은 사용자만 결정할 수 있는 항목. 나머지는 진행하며 결정
- 설계를 `_workspaces/{slug}/design.md`로 문서화 — 요구사항·경계(YAGNI)·제외 범위·성공 기준·검토된 대안 포함. 사용자 승인 후 착수

## 3. TDD 사이클

> 철칙: 실패하는 테스트 없이는 스킬도 없음 (`writing-skills` 참조)

- **RED** — 스킬 없이 베이스라인 서브에이전트를 실행하고 실패 행동을 그대로 실측 기록. 예: which-way-should-i-go의 baseline은 세부 질문 선행, 학습 지식만으로 비교, "병렬화 부적합" 자기 기각을 보임
- **GREEN** — 실측된 실패만 직격하는 최소 스킬 작성 후, 동일 시나리오 재실행으로 준수 확인
- **REFACTOR** — 경계 테스트(트리거되어야 하는 케이스 + 트리거되면 안 되는 오탐 방지 케이스의 대조쌍) 실행. 허점이 없으면 본문 무수정

핵심: baseline의 합리화 문구가 그대로 스킬의 "흔한 실수" 표 재료가 됨.

## 4. 내용 추가 반복

기능 추가(게이트·변형 규칙 등)마다 **편집 → 대조 시나리오 테스트 → 원자 커밋**을 반복함.

- 테스트는 항상 쌍으로 구성: 새 규칙이 발동해야 하는 케이스 + 발동하면 안 되는 케이스 (예: 신생 시장 vs 성숙 도메인)
- "테스트 없이 스킬 편집 = 철칙 위반" — 사소한 추가에도 예외 없음

## 5. 문서 정합성(SSOT) 확인

- **상위 문서 동기화** — CLAUDE.md의 스킬 개수·파일 트리 등 집계성 기술 갱신
- **참조 전 실물 확인** — 다른 스킬을 참조하기 전에 실제 내용을 읽어 차용 가능한지 검증. 이름만 보고 링크하지 않음
- **배포 환경 의존성 제거** — 배포받은 환경에 존재하지 않을 수 있는 외부 스킬·도구 참조를 본문에서 제거하고 취지만 흡수
- **불일치 발견 경로는 2가지** — 정적 대조(writing-skills 체크리스트)와 동작 테스트. description(5+ files)과 본문(3개 이상)의 발동 임계값 불일치는 동작 테스트에서 에이전트가 실제 혼란을 보고하며 확정됨

## 6. 의존 스킬 역점검

새 스킬이 참조하는 기존 스킬도 같은 방법론으로 점검함:

- 정적 점검 — frontmatter, CSO 규칙(description은 트리거만, 워크플로우 요약 금지), SSOT
- 동작 테스트 — 압박 내성(시간·매몰 비용·승인 암시), 결함을 심은 입력의 표면화 능력
- **테스트로 실증된 공백만 수정** — 세심하게 튜닝된 본문(질문 카테고리, Golden Rules, Red Flags 표)은 증거 없이 변경 금지. 수정 후 재검증

## 7. 테스트 자산화

개발 중 사용한 실측 테스트를 `tests/pressure-scenarios/` 형식(시나리오 프롬프트 + baseline 실패 + 기대 동작)으로 보존함:

- baseline이 "예상"이 아닌 **실측**임을 관찰 날짜와 함께 명시 — "증거 없이 변경 금지" 지침의 증거 자료 역할
- 점검에서 수정된 항목은 **회귀 테스트 겸용**으로 표기 — 향후 개정 시 되돌아가기 쉬운 지점을 고정
- 스위트 README 목록 갱신

## 8. 커밋 규율

- 단계별 원자 커밋 — 유형 구분(feat 신설·기능 추가 / refactor 다듬기 / fix 결함 수정 / test 자산화)
- 설계 문서(`_workspaces/`)는 gitignore 대상이므로 커밋에 포함되지 않음 — 근거가 필요한 내용은 커밋 메시지 본문과 tests/에 남김

## 핵심 패턴 요약

1. **원자 단위** — 모든 변경은 "편집 + 대조쌍 테스트 + 커밋"의 원자 단위로 수행
2. **증거 사슬** — 실측 baseline → 스킬 본문("흔한 실수" 표) → tests/ 회귀 시나리오로 이어지는 증거 사슬 유지
3. **양방향 정합성** — 참조하는 문서는 참조 전에 실물을 읽고, 참조된 문서도 역방향으로 점검
